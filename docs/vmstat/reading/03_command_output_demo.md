# vmstat 命令输出实战解析

> ★ 每个命令都实际执行，展示真实输出并逐列/逐行注解。
> 环境：4 核 CPU / 7.3 GiB 内存 / /dev/vda + /dev/vdb (virtio-blk)
> 系统：procps-ng 4.0.4

---

## 场景 1：vmstat 1 5 — 基础输出逐列注解

### 命令

```bash
vmstat 1 5
```

### 真实输出

```
procs -----------memory---------- ---swap-- -----io---- -system-- -------cpu-------
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st gu
 0  0      0 1103476  50268 3771464    0    0    17   175 1200    5  1  0 98  0  0  0
 1  0      0 1102640  50268 3771480    0    0     0     0 5130 7086 19  4 77  0  0  0
 1  0      0 1102508  50276 3771576    0    0    32     0 4919 6587 18  3 78  0  0  0
 3  0      0 1101500  50276 3771560    0    0     0    24 4092 5727 14  3 84  0  0  0
 1  0      0 1102008  50276 3771560    0    0     0     0 3884 5429 13  3 84  0  0  0
```

### 逐列注解

#### procs 区域

| 列 | 第一行值 | 含义                                  | 解读                          |
|---|---------|---------------------------------------|------------------------------|
| `r` | 0       | 可运行进程数（含正在执行+等待调度）         | ★ 0 表示 CPU 完全空闲          |
| `b` | 0       | 不可中断睡眠的进程数（通常在等 I/O）        | ★ 0 表示无 I/O 阻塞           |

> ★ **第一行是启动以来的平均值**，第二行起才是真实瞬时值。
> 第一行的 `r=0` 不代表系统空闲——它被长期 idle 时间平均掉了。

#### memory 区域

| 列      | 第一行值  | 含义                         | 解读                                       |
|--------|---------|-----------------------------|------------------------------------------|
| `swpd` | 0       | 已使用的 swap 空间 (KiB)       | ★ 0 表示未使用 swap（本机未配置交换分区）        |
| `free` | 1103476 | 空闲物理内存 (KiB) ≈ 1078 MiB  | 约 1 GiB 空闲                               |
| `buff` | 50268   | 块设备缓冲区 (KiB) ≈ 49 MiB    | 用于 raw block I/O 的 buffer_head           |
| `cache`| 3771464 | 页面缓存 (KiB) ≈ 3683 MiB     | ★ 最大一块！约 3.6 GiB 被文件缓存占用          |

> ★ Linux 内存管理的核心哲学：**空闲内存 = 浪费内存**。
> `free` 小不代表内存紧张——`buff + cache` 在需要时会立即回收。
> 真正可用内存 ≈ `free + buff + cache`（约 4.7 GiB）。

#### swap 区域

| 列  | 第一行值 | 含义                      | 解读                                 |
|----|---------|--------------------------|-------------------------------------|
| `si`| 0       | 从 swap 换入 (KiB/s)      | ★ 持续非零 = 内存压力，需要换回页面        |
| `so`| 0       | 向 swap 换出 (KiB/s)      | ★ 持续非零 = 内存不足，正在写入交换分区      |

> 本系统未配置 swap（`total swap = 0`），所以 si/so 始终为 0。

#### io 区域

| 列  | 第一行值 | 含义                       | 解读                                      |
|----|---------|--------------------------|------------------------------------------|
| `bi`| 17      | 从块设备读入 (blocks/s)     | ★ 约 17 KiB/s 读磁盘，系统启动以来的平均值    |
| `bo`| 175     | 向块设备写出 (blocks/s)     | ★ 约 175 KiB/s 写磁盘，主要是日志和缓存刷新   |

> ★ `bi`/`bo` 单位是 **blocks/s**（通常 1 block = 1 KiB），不是 KB/s。
> 第二行起：`bi=0, bo=0` → 该采样周期内无磁盘 I/O。

#### system 区域

| 列  | 第一行值 | 含义                  | 解读                                         |
|----|---------|----------------------|---------------------------------------------|
| `in`| 1200    | 中断数 (次/秒)         | ★ 1200 是平均值；瞬时值 3884~5130（第二行起）      |
| `cs`| 5       | 上下文切换数 (次/秒)    | ★ 第一行 `cs=5` 是平均值的极端体现；瞬时值 5429~7086 |

> ★ **第一行的 in/cs 是最具误导性的数据！**
> `in=1200, cs=5` 是启动以来的总中断/总切换 / 总秒数 = 极低的平均值。
> 实际每秒中断约 4000~5000 次，上下文切换约 5000~7000 次。

#### cpu 区域

| 列  | 第一行值 | 含义                        | 解读                                   |
|----|---------|----------------------------|---------------------------------------|
| `us`| 1       | 用户态 CPU 百分比              | ★ 1% 用户态（被长期 idle 平均）          |
| `sy`| 0       | 内核态 CPU 百分比              | 接近 0%                                |
| `id`| 98      | 空闲 CPU 百分比                | ★ 98% 空闲——启动以来平均值              |
| `wa`| 0       | I/O 等待 CPU 百分比            | 0% 无 I/O 等待                          |
| `st`| 0       | 被宿主机偷取的 CPU 百分比       | 0%（非虚拟化或宿主机未争抢）              |
| `gu`| 0       | 客户机使用的 CPU 百分比         | 0%（无 KVM 客户机运行）                  |

> ★ 第二行起的瞬时值：`us=19, sy=4, id=77` → 说明该秒内有 23% 的 CPU 被使用。
> 这与第一行的 `id=98` 形成鲜明对比——**再次印证第一行是平均值**。

### 逐行注解

#### 内存统计

| 行                              | 值           | 注解                                              |
|--------------------------------|-------------|--------------------------------------------------|
| `total memory`                 | 7638924 K   | ★ 物理内存总量 ≈ 7.3 GiB                            |
| `used memory`                  | 3762908 K   | 已使用内存 ≈ 3.6 GiB                                |
| `active memory`                | 2745908 K   | 活跃页面 ≈ 2.6 GiB（近期被访问，内核倾向保留）           |
| `inactive memory`              | 3326576 K   | 不活跃页面 ≈ 3.2 GiB（长时间未访问，可回收）             |
| `free memory`                  | 1102336 K   | ★ 完全空闲 ≈ 1.05 GiB                               |
| `buffer memory`                | 50276 K     | 块设备 buffer ≈ 49 MiB                              |
| `swap cache`                   | 3771560 K   | ★ 已换出但仍在 page cache 中的页面 ≈ 3.6 GiB          |
| `total/used/free swap`         | 全部 0       | ★ 未配置交换分区                                     |

> ★ **内存平衡公式：** total = used + free + buffer + swap cache
> 7638924 ≈ 3762908 + 1102336 + 50276 + 3771560 = 8687080（差异来自内核 slab 等）
> active + inactive ≈ used + buffer + swap cache（不严格相等，有重叠计算）

#### CPU ticks 统计

| 行                    | 值            | 注解                                          |
|----------------------|--------------|----------------------------------------------|
| `non-nice user`      | 8,313,875    | 用户态（普通优先级）tick 数                          |
| `nice user`          | 71,731       | 用户态（低优先级 nice > 0）tick 数                   |
| `system`             | 1,865,773    | 内核态 tick 数                                   |
| `idle`               | 702,119,699  | ★ 空闲 tick 数——远大于其他，系统大部分时间在空闲         |
| `IO-wait`            | 208,355      | I/O 等待 tick 数                                 |
| `IRQ`                | 1,392,759    | 硬中断处理 tick 数                                 |
| `softirq`            | 280,935      | 软中断处理 tick 数                                 |
| `stolen`             | 0            | 被宿主机偷取的 tick 数（虚拟化环境才有值）              |
| `guest`              | 0            | KVM 客户机使用的 tick 数                            |

> ★ **CPU 使用率计算：**
> 总 ticks = 8313875 + 71731 + 1865773 + 702119699 + 208355 + 1392759 + 280935 = 714,253,127
> idle 占比 = 702119699 / 714253127 ≈ **98.3%**
> 系统运行时间绝大部分都在空闲。

#### I/O 与系统统计

| 行                       | 值               | 注解                                     |
|-------------------------|-----------------|----------------------------------------|
| `paged in`              | 30,824,458 K    | 从磁盘读入的页面总量 ≈ 29.4 GiB              |
| `paged out`             | 314,093,676 K   | ★ 写回磁盘的页面总量 ≈ 299.6 GiB            |
| `pages swapped in/out`  | 0               | 未发生 swap                               |

## 场景 4：配合 fio 压力测试观察 vmstat 变化

### 命令

```bash
# 后台启动 vmstat 采集 8 秒
vmstat 1 8 > /tmp/vmstat_fio.txt &
sleep 1
# 启动 fio 随机写压力测试：2 个 job，iodepth=32，4K 块，运行 6 秒
fio --name=pressure_test --ioengine=libaio --direct=1 --rw=randwrite --bs=4k \
    --filename=/dev/vdb --runtime=6 --time_based --iodepth=32 --numjobs=2 \
    --group_reporting
wait
```

### fio 结果摘要

```
write: IOPS=18.8k, BW=73.3MiB/s (76.9MB/s)(440MiB/6005msec)
  clat (usec): min=134, max=12500, avg=3395.36
  lat (usec) : min=189, max=12503, avg=3407.88
```

### vmstat 输出（fio 运行期间）

```
 1  1      0 1086036  50328 3771384    0    0    16   260 4594 6582 16  3 82  0  0  0
 0  0      0 1079236  51816 3775968    0    0  4824 114216 31728 43248 21 14 63  1  0  0
 3  0      0 1078252  51816 3775968    0    0     0 64000 20884 27308 20 10 70  0  0  0
 1  0      0 1065032  51816 3775964    0    0     0 64000 20909 28217 27 15 58  0  0  0
 1  0      0 1072816  51816 3776068    0    0     0 63996 20739 27160 20  9 71  0  0  0

 1  0      0 1073812  51828 3776068    0    0     0 63996 20206 26344 13  7 81  0  0  0
```

### 逐阶段注解

#### 阶段 1：fio 启动前（第 1-2 行）

```
r=0, b=0, bo=175~260, in=1200~4594, cs=5~6582, us=1~16%, id=82~98%
```
- ★ 系统空闲状态，`bo` 只有几百 blocks/s
- 中断和上下文切换是正常后台水平

#### 阶段 2：fio 全力运行（第 3-6 行）★ 核心观察

```
bo = 64000~114216 blocks/s     → 约 62~111 MiB/s 写出，与 fio 的 BW=73.3MiB/s 吻合
                                 （fio 使用 Direct I/O 绕过 cache，vmstat 的 bo 含元数据写入）
in = 20000~31000 次/s          → ★ 中断飙升至 3 万！每个 I/O 完成触发一次中断
cs = 26000~43000 次/s          → ★ 上下文切换飙升 4~6 倍（正常约 5000~7000）
us = 17~27%, sy = 8~15%       → ★ CPU 使用率从 1% 飙升到 30~40%
wa = 0~1%                     → ★ wa 几乎为零！因为 libaio + Direct I/O 不阻塞
```

> ★ **关键发现：** fio 使用 `ioengine=libaio` + `direct=1` 时，I/O 是异步的，
> 进程不会进入 D 状态（不可中断睡眠），所以 **`b` 列几乎为 0，`wa` 也很低**。
> 如果用 `sync` 引擎，会看到 `b > 0` 且 `wa` 显著升高。

#### 阶段 3：fio 结束后（第 7-8 行）

```
bo 仍然 ≈ 64000              → ★ 缓存回写仍在进行（fio 已停，但脏页还在刷盘）
in/cs/us 开始下降              → 系统正在恢复空闲状态
```

### fio vs vmstat 数据对照

| 指标          | fio 报告               | vmstat 观测                    | 一致性  |
|--------------|----------------------|-------------------------------|--------|
| 写入带宽       | 73.3 MiB/s           | bo ≈ 64000 blocks/s ≈ 62 MiB/s | ★ 近似  |
| IOPS          | 18,800               | 无法直接得到（vmstat 不区分 I/O 大小） | -      |
| 延迟          | avg 3.4ms            | 无法直接得到（vmstat 无延迟数据）       | -      |
| CPU 开销      | 未报告                 | us+sy ≈ 30~40%               | ★ 补充  |

> ★ **vmstat 的价值：** fio 不报告系统级开销，vmstat 填补了 CPU 使用率、中断、上下文切换的空白。

## 场景 5：诊断 CPU 高但吞吐低（关注 cs/in/r 列）

### 场景构造

```bash
# 快速 fork 200 个短生命周期进程，模拟高上下文切换
vmstat 1 5 > /tmp/vmstat_cs.txt &
sleep 1
for i in $(seq 1 200); do /bin/true & done
wait
```

### 真实输出

```
procs -----------memory---------- ---swap-- -----io---- -system-- -------cpu-------
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st gu
 1  0      0 1064000  51828 3787888    0    0    17   175 1200    5  1  0 98  0  0  0
 0  1      0 1060060  51832 3788052    0    0     4     0 4926 6209 22  6 72  0  0  0
 0  0      0 1055648  51832 3788212    0    0    84     0 5217 6888 17 10 73  0  0  0
 5  0      0 1055648  51840 3788208    0    0     0    52 4582 6557 16  3 82  0  0  0
 1  0      0 1061500  51840 3788212    0    0     0     0 5279 5768 29  4 68  0  0  0
```

### 关键列分析

| 列  | 值范围       | 诊断意义                                            |
|----|------------|----------------------------------------------------|
| `r` | 1~5        | ★ 第 4 行 `r=5` 超过 CPU 核心数（4 核），说明有进程在排队等待  |
| `cs`| 5768~6888  | 上下文切换偏高（正常空闲约 5000），fork 200 个进程导致          |
| `in`| 4582~5279  | 中断数略有升高（进程创建/退出触发信号和中断）                    |
| `sy`| 3~10%      | ★ 系统态占比升高——fork/exec 都是系统调用密集操作              |
| `b` | 0~1        | 有 1 个进程在 I/O 等待（进程退出时的文件操作）                  |

> ★ **诊断模式总结：**
> - `r > CPU核心数` → CPU 资源瓶颈，进程在排队
> - `cs > 20000` → 过多上下文切换（锁竞争、过多短任务）
> - `sy > us` → 系统调用开销大（可能是大量小 I/O 或频繁 fork）
> - `in > 50000` → 中断风暴（网络包风暴或 I/O 中断）

### cs/in/r 三列联合诊断矩阵

```
          cs 低        cs 中          cs 高
r 低    │ 系统空闲    │ 正常 I/O     │ 锁竞争/中断风暴
in 低   │ ★ 理想状态  │ 少量 I/O     │ spinlock 竞争
        │            │              │
r 高    │ 罕见        │ CPU 密集     │ ★ 最差情况
in 高   │            │ 网络/I/O密集  │ CPU+I/O 双重瓶颈
```

## 场景 6：各列诊断阈值速查表

### 基于本机环境（4 核 / 7.3 GiB）的参考阈值

> ★ 以下阈值为经验参考值，实际基准因硬件配置和负载模式而异。

#### procs 区域

| 列  | 🟢 正常         | 🟡 注意            | 🔴 危险              |
|----|----------------|-------------------|---------------------|
| `r`| 0 ~ CPU 核心数   | > CPU 核心数        | > 2× CPU 核心数      |
| `b`| 0              | 1~3               | > 5（I/O 严重阻塞）    |

> 本机 4 核：`r ≤ 4` 正常，`r > 4` 注意，`r > 8` 危险。

#### memory 区域

| 列       | 🟢 正常              | 🟡 注意              | 🔴 危险                  |
|---------|---------------------|---------------------|-------------------------|
| `swpd`  | 0 或很小              | > 总内存 10%          | > 总内存 50%              |
| `free`  | > 总内存 20%          | < 总内存 10%          | < 100 MiB               |

> 本机 7.3 GiB：`free > 1.5G` 正常，`free < 730M` 注意，`free < 100M` 危险。

#### swap 区域

| 列  | 🟢 正常       | 🟡 注意         | 🔴 危险              |
|----|-------------|----------------|---------------------|
| `si`| 0           | 1~100 KiB/s    | > 1000 KiB/s        |
| `so`| 0           | 1~100 KiB/s    | > 1000 KiB/s        |

> ★ `si`/`so` 持续非零就是警告信号——说明物理内存不足，正在使用 swap。

#### io 区域

| 列  | 🟢 正常         | 🟡 注意            | 🔴 危险                 |
|----|----------------|-------------------|------------------------|
| `bi`| 0 ~ 10000     | 10000 ~ 50000     | > 50000 (≈50 MiB/s 读)  |
| `bo`| 0 ~ 10000     | 10000 ~ 50000     | > 50000 (≈50 MiB/s 写)  |

> ★ bi/bo 高本身不是问题——需结合 `wa` 和 `b` 判断是否构成瓶颈。

#### system 区域

| 列  | 🟢 正常         | 🟡 注意            | 🔴 危险               |
|----|----------------|-------------------|----------------------|
| `in`| < 10000        | 10000 ~ 50000     | > 50000（中断风暴）      |
| `cs`| < 10000        | 10000 ~ 50000     | > 50000（过度切换）      |

> ★ cs 过高通常意味着：
> - 应用层锁竞争激烈
> - 过多短生命周期进程/线程
> - 中断负载均衡不当

#### cpu 区域

| 列  | 🟢 正常      | 🟡 注意        | 🔴 危险             |
|----|------------|---------------|--------------------|
| `us`| 0 ~ 70%    | 70 ~ 90%      | > 90%（CPU 饱和）     |
| `sy`| 0 ~ 20%    | 20 ~ 40%      | > 40%（内核开销过大）   |
| `id`| > 30%      | 10 ~ 30%      | < 10%              |
| `wa`| 0 ~ 5%     | 5 ~ 20%       | > 20%（I/O 瓶颈）     |
| `st`| 0          | 1 ~ 10%       | > 10%（宿主机争抢）    |

> ★ `us + sy > 90%` 且 `id < 5%` → CPU 已完全饱和。
> ★ `wa > 20%` → 需进一步用 `iostat -x` 或 `blktrace` 定位具体磁盘。
> ★ `st > 0` → 虚拟机环境，宿主机资源争抢，需联系基础设施团队。

## 快速诊断流程

当系统出现性能问题时，按以下顺序使用 vmstat：

```
Step 1: vmstat 1 5
        ↓
  ┌─ r > CPU 核心数？
  │   → YES: CPU 瓶颈 → 用 top 找进程
  │   → NO:  ↓
  │
  ├─ b > 0 持续？
  │   → YES: I/O 阻塞 → vmstat -d 看哪块盘
  │   → NO:  ↓
  │
  ├─ wa > 10%？
  │   → YES: I/O 等待 → iostat -x 定位设备
  │   → NO:  ↓
  │
  ├─ si/so > 0？
  │   → YES: 内存不足 → free -h 确认可用内存
  │   → NO:  ↓
  │
  ├─ cs > 50000？
  │   → YES: 锁竞争/过多短任务 → perf record 找热点
  │   → NO:  ↓
  │
  └─ st > 0？
      → YES: 宿主机争抢 → 联系基础设施团队
      → NO: 可能不是系统层问题，检查应用层
```
