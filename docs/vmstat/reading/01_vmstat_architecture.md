# vmstat 架构与数据流全景

## 1. vmstat 是什么？

`vmstat`（Virtual Memory Statistics）是 Linux 最经典的系统级性能概览工具。
它从内核的 `/proc` 伪文件系统中读取统计数据，以固定间隔刷新，输出一个
紧凑的表格，涵盖**进程、内存、交换、I/O、系统、CPU**六大维度。

```
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2  0      0 123456  56789 890123    0    0    12    34  567  890  5  2 92  1  0
```

**★ 核心价值：** 一行输出就能判断系统是 CPU 瓶颈、内存瓶颈还是 I/O 瓶颈。


## 2. 六组输出列详解

### 2.1 procs — 进程队列

```
procs
 r  b
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `r` | 运行队列中的进程数（可运行 + 正在运行） | 个 | `/proc/stat` 的 `procs_running` |
| `b` | 不可中断睡眠的进程数（通常在等 I/O） | 个 | `/proc/stat` 的 `procs_blocked` |

**★ 关键理解：**
- `r` 包含**正在 CPU 上执行的进程** + **等待 CPU 的进程**
- 如果 `r > CPU 核数`，说明有进程在排队等 CPU → **CPU 过载**
- `b > 0` 持续出现 → 进程被 I/O 阻塞 → **I/O 瓶颈**

```
                    ┌─────────────────────────────────┐
                    │         进程状态流转              │
                    ├─────────────────────────────────┤
                    │                                 │
                    │   TASK_RUNNING ──── 正在执行     │
                    │        │                        │
                    │        │ CPU 被抢占              │
                    │        ▼                        │
                    │   运行队列 (r)                   │
                    │        │                        │
                    │        │ 获得 CPU                │
                    │        ▼                        │
                    │   TASK_RUNNING ──── 正在执行     │
                    │                                 │
                    │   TASK_UNINTERRUPTIBLE           │
                    │        │                        │
                    │        │ 等待 I/O                │
                    │        ▼                        │
                    │   阻塞队列 (b)                   │
                    └─────────────────────────────────┘
```


### 2.2 memory — 内存使用

```
-----------memory----------
   swpd   free   buff  cache
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `swpd` | 已使用的 swap 空间 | KB | `/proc/meminfo` 的 `SwapTotal - SwapFree` |
| `free` | 空闲物理内存 | KB | `/proc/meminfo` 的 `MemFree` |
| `buff` | 用于块设备 I/O 的缓冲区 | KB | `/proc/meminfo` 的 `Buffers` |
| `cache` | 页面缓存（文件内容缓存） | KB | `/proc/meminfo` 的 `Cached` |

**★ 注意：**
- vmstat 的内存列**不包含** `slab`、`page tables`、`kernel stacks` 等内核内存
- `free` 很低不一定是问题——Linux 会积极用空闲内存做缓存
- 真正判断内存是否充足，要看 `si/so`（swap 活动）和 `/proc/meminfo` 的 `MemAvailable`

```
物理内存分配全景：

┌──────────────────────────────────────────────┐
│              物理内存总量                      │
├──────────┬──────────┬───────┬────────────────┤
│  用户进程  │  page    │ buff  │  内核使用        │
│  (匿名页)  │  cache   │       │  (slab/栈/页表)  │
├──────────┴──────────┴───────┴────────────────┤
│                    free                       │
└──────────────────────────────────────────────┘
           ▲                    ▲
           │                    │
     vmstat 不直接显示      vmstat 显示的
                          free/buff/cache
```


### 2.3 swap — 交换活动

```
---swap--
 si   so
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `si` | 每秒从 swap 换入内存的数据量 | KB/s | `/proc/vmstat` 的 `pswpin` 差值 × 页大小 / 间隔 |
| `so` | 每秒从内存换出到 swap 的数据量 | KB/s | `/proc/vmstat` 的 `pswpout` 差值 × 页大小 / 间隔 |

**★ 关键理解：**
- `si > 0` 说明系统正在从磁盘读回之前换出的页面 → 应用会感受到延迟
- `so > 0` 说明系统正在把内存页面写到磁盘 → 内存压力大
- 任何持续的 `si/so > 0` 都是**内存不足**的信号
- 偶尔的 swap 活动可能是正常的（比如冷页面回收），持续活动才是问题


### 2.4 io — 块设备 I/O

```
-----io----
  bi    bo
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `bi` | 每秒从块设备读入的数据量 | blocks/s（通常 1K/block） | `/proc/vmstat` 的 `pgpgin` 差值 / 间隔 |
| `bo` | 每秒写出到块设备的数据量 | blocks/s（通常 1K/block） | `/proc/vmstat` 的 `pgpgout` 差值 / 间隔 |

**★ 注意：**
- `bi/bo` 的单位是 **blocks/s**，在大多数系统上 1 block = 1 KB
- 这些值来自内核的页面级统计（page I/O），不是设备级统计
- 如果需要设备级 I/O 统计，应该用 `iostat` 或 `blktrace`
- `bi` 很高 → 大量读磁盘（可能是缓存未命中）
- `bo` 很高 → 大量写磁盘（可能是日志/数据库写入密集）


### 2.5 system — 系统活动

```
-system--
  in   cs
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `in` | 每秒中断次数（硬件中断 + 软中断） | 次/s | `/proc/stat` 的 `intr` 差值 / 间隔 |
| `cs` | 每秒上下文切换次数 | 次/s | `/proc/stat` 的 `ctxt` 差值 / 间隔 |

**★ 关键理解：**
- `in`（中断）高 → 设备频繁触发中断（网络包、磁盘 I/O 完成等）
- `cs`（上下文切换）高 → 进程/线程频繁切换 CPU，调度开销大
- 高 `cs` 的常见原因：
  - 线程过多，竞争激烈
  - 锁竞争导致频繁阻塞/唤醒
  - 大量短生命周期的进程创建
- 中断和上下文切换本身不是坏事，但**过高的值**意味着 CPU 在做大量"管理工作"而非实际计算


### 2.6 cpu — CPU 时间分配

```
------cpu-----
us sy id wa st
```

| 列 | 含义 | 单位 | 数据来源 |
|----|------|------|----------|
| `us` | 用户态 CPU 时间（不含 nice） | % | `/proc/stat` 的 `user` 字段 |
| `sy` | 内核态 CPU 时间 | % | `/proc/stat` 的 `system` 字段 |
| `id` | 空闲时间（无进程等待 I/O） | % | `/proc/stat` 的 `idle` 字段 |
| `wa` | 等待 I/O 完成的时间 | % | `/proc/stat` 的 `iowait` 字段 |
| `st` | 被虚拟机管理器偷取的时间 | % | `/proc/stat` 的 `steal` 字段 |

**★ 注意：**
- 所有百分比加起来 = 100%（在多核系统上是所有核心的平均值）
- `us + sy` 高 → CPU 忙于执行工作
- `wa` 高 → CPU 在等 I/O，说明瓶颈在存储/网络
- `st` 只在虚拟化环境中非零（AWS EC2、KVM 等）
- `id` 接近 100% → 系统基本空闲


## 3. /proc/vmstat 关键指标详解

vmstat 工具读取 `/proc/vmstat` 获取内存和 I/O 相关的原始计数器。
这些计数器是**自启动以来的累计值**，vmstat 通过两次采样的差值来计算速率。

### 3.1 页面 I/O 指标

| 指标 | 含义 | vmstat 对应列 |
|------|------|---------------|
| `pgpgin` | 页面读入总量（从磁盘到内存） | `bi`（除以间隔） |
| `pgpgout` | 页面写出总量（从内存到磁盘） | `bo`（除以间隔） |
| `pswpin` | swap 页面换入总量 | `si`（乘以页大小，除以间隔） |
| `pswpout` | swap 页面换出总量 | `so`（乘以页大小，除以间隔） |

```
页面 I/O 数据流：

  磁盘                          内存
  ┌───┐    pgpgin (bi)         ┌───────┐
  │   │ ─────────────────────► │       │
  │   │                        │ page  │
  │   │ ◄───────────────────── │ cache │
  └───┘    pgpgout (bo)        └───────┘

  swap 分区                      匿名页
  ┌───┐    pswpin (si)         ┌───────┐
  │   │ ─────────────────────► │       │
  │   │                        │ anon  │
  │   │ ◄───────────────────── │ pages │
  └───┘    pswpout (so)        └───────┘
```


### 3.2 页面回收指标

| 指标 | 含义 | 诊断价值 |
|------|------|----------|
| `pgscan_kswapd` | kswapd 扫描的页面数 | 高值 → 内存压力，kswapd 频繁运行 |
| `pgscan_direct` | 直接回收扫描的页面数 | ★ 非零 → 严重内存压力，进程被阻塞等待内存 |
| `pgsteal_kswapd` | kswapd 回收的页面数 | 与 pgscan 的比值 = 回收效率 |
| `pgsteal_direct` | 直接回收的页面数 | 同上 |
| `pgfault` | 页面错误总数（包括软错误） | 高值不一定有问题（mmap 首次访问也会触发） |
| `pgmajfault` | 主要页面错误（需要磁盘 I/O） | ★ 高值 → 频繁的磁盘读取，性能瓶颈 |

**★ 关键理解：**

```
页面回收路径：

  内存分配请求
       │
       ▼
  有足够空闲页？── 是 ──► 直接分配（快速路径）
       │
       否
       │
       ▼
  kswapd 后台回收（pgscan_kswapd / pgsteal_kswapd）
       │
       ▼
  回收足够？── 是 ──► 分配成功
       │
       否
       │
       ▼
  ★ 直接回收（pgscan_direct / pgsteal_direct）
       │                          ▲
       │              进程在此阻塞等待！
       │              这是性能杀手
       ▼
  分配成功（但延迟很高）
```

- `pgscan_direct > 0` → 进程的内存分配被阻塞，等待页面回收完成
- 回收效率 = `pgsteal / pgscan`，接近 100% 说明扫描的页面大多可以回收
- 回收效率低 → 大量不可回收的页面（如内核 slab），可能需要排查内核内存泄漏


### 3.3 OOM 与脏页指标

| 指标 | 含义 | 诊断价值 |
|------|------|----------|
| `oom_kill` | OOM killer 触发次数 | ★ 非零 → 系统曾因内存不足杀死进程 |
| `nr_dirty` | 当前脏页数量（已修改但未写回） | 高值 → 写 I/O 积压 |
| `nr_writeback` | 当前正在写回的页面数 | 高值 → 磁盘写入带宽不足 |
| `nr_slab_reclaimable` | 可回收的 slab 内存 | 高值 → dentry/inode 缓存占用大量内存 |
| `nr_slab_unreclaimable` | 不可回收的 slab 内存 | 高值 → 可能的内核内存泄漏 |

**★ OOM 排查链路：**
```
oom_kill > 0
    │
    ▼
检查 /var/log/messages 或 dmesg
    │
    ▼
找到被杀的进程和当时的内存快照
    │
    ▼
分析是内存泄漏还是配置不足
```


## 4. /proc/stat CPU 时间字段映射

vmstat 的 CPU 百分比来自 `/proc/stat` 的第一行：

```
$ head -1 /proc/stat
cpu  123456 7890 45678 8901234 56789 0 12345 0 0 0
```

### 4.1 字段定义

```
cpu  user nice system idle iowait irq softirq steal guest guest_nice
      │    │     │      │      │    │      │       │      │
      │    │     │      │      │    │      │       │      └── 客户机 nice 时间
      │    │     │      │      │    │      │       └── 客户机时间（已被 user 包含）
      │    │     │      │      │    │      └── 被 hypervisor 偷取的时间
      │    │     │      │      │    └── 软中断处理时间
      │    │     │      │      └── 硬件中断处理时间
      │    │     │      └── 空闲时间
      │    │     └── 内核态时间
      │    └── nice 调整后的用户态时间
      └── 普通用户态时间
```

### 4.2 映射关系

| /proc/stat 字段 | vmstat 列 | 计算公式 |
|-----------------|-----------|----------|
| `user` | `us` | `user / total × 100` |
| `nice` | （合入 `us`） | 部分实现合入 us，部分单独显示 |
| `system` | `sy` | `(system + irq + softirq) / total × 100` |
| `idle` | `id` | `idle / total × 100` |
| `iowait` | `wa` | `iowait / total × 100` |
| `steal` | `st` | `steal / total × 100` |

其中 `total = user + nice + system + idle + iowait + irq + softirq + steal`

**★ 关键理解：**
- `sy` 包含了 `irq` 和 `softirq`——高中断负载会表现为高 `sy`
- `iowait` 是一种特殊的"空闲"——CPU 没有可执行的进程，但有进程在等 I/O
- `guest` 和 `guest_nice` 已经被 `user` 和 `nice` 包含，不重复计算
- 在虚拟化环境中，`steal` 表示宿主机把 CPU 时间分给了其他虚拟机

### 4.3 iowait 的陷阱

```
                    ┌───────────────────────────────────┐
                    │     iowait 的常见误解              │
                    ├───────────────────────────────────┤
                    │                                   │
                    │  ✗ "iowait 高 = I/O 慢"           │
                    │  ✓ iowait 高 = CPU 空闲且          │
                    │    有进程在等 I/O                  │
                    │                                   │
                    │  ✗ "iowait 是 CPU 在忙"           │
                    │  ✓ iowait 期间 CPU 其实是空闲的    │
                    │    只是标记为"有 I/O 等待"         │
                    │                                   │
                    │  ★ 如果同时有高 us/sy 和 I/O       │
                    │    iowait 会被掩盖（CPU 忙于       │
                    │    计算，不算 iowait）             │
                    └───────────────────────────────────┘
```


## 5. 差值计算原理

vmstat 不是直接显示瞬时值，而是通过**两次采样的差值除以时间间隔**来计算速率。

### 5.1 计算公式

```
                第二次采样值 - 第一次采样值
  速率 = ─────────────────────────────────────
                     时间间隔（秒）
```

**示例：计算 bi（块设备读入）**

```
第 1 次采样（T=0s）：pgpgin = 1000000
第 2 次采样（T=1s）：pgpgin = 1000012

bi = (1000012 - 1000000) / 1 = 12 blocks/s
```

### 5.2 采样流程

```
vmstat 1（每秒采样一次）的执行流程：

T=0s                          T=1s                          T=2s
 │                             │                             │
 ▼                             ▼                             ▼
┌──────────┐              ┌──────────┐              ┌──────────┐
│ 第 1 次   │              │ 第 2 次   │              │ 第 3 次   │
│ 读取      │              │ 读取      │              │ 读取      │
│ /proc/*  │              │ /proc/*  │              │ /proc/*  │
└────┬─────┘              └────┬─────┘              └────┬─────┘
     │                         │                         │
     │    差值 = S2 - S1       │    差值 = S3 - S2       │
     │    速率 = 差值 / 1s     │    速率 = 差值 / 1s     │
     │                         │                         │
     ▼                         ▼                         ▼
┌──────────┐              ┌──────────┐              ┌──────────┐
│ 输出第 1  │              │ 输出第 2  │              │ 输出第 3  │
│ 行数据    │              │ 行数据    │              │ 行数据    │
└──────────┘              └──────────┘              └──────────┘
```

### 5.3 首次输出的特殊性

```
$ vmstat 1 3
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 1  0 123456 789012  34567 456789    0    0     5    10  100  200  3  1 95  1  0  ← 第 1 行
 2  1 123456 788900  34567 456800    0    4    20    50  300  500 10  5 80  5  0  ← 第 2 行
 1  0 123456 789100  34567 456780    0    0     2     8  150  250  4  2 93  1  0  ← 第 3 行
```

**★ 第 1 行 vs 后续行：**
- 第 1 行的速率列（si/so/bi/bo/in/cs/us/sy/id/wa/st）是**自系统启动以来的平均值**
- 从第 2 行开始才是真正的"每秒"数据
- 这是 vmstat 的一个设计特点，分析时应**跳过第 1 行**

### 5.4 不同列的计算方式

| 列类型 | 计算方式 | 示例 |
|--------|----------|------|
| 累计计数器 | 差值 / 时间间隔 | `bi`, `bo`, `si`, `so`, `in`, `cs` |
| 瞬时值 | 直接读取，无需差值 | `r`, `b`, `swpd`, `free`, `buff`, `cache` |
| CPU 百分比 | 差值 / 总 CPU 时间差值 × 100 | `us`, `sy`, `id`, `wa`, `st` |

```
CPU 百分比计算示例：

T=0s: cpu  user=1000 nice=10 system=200 idle=8000 iowait=50 irq=10 softirq=20 steal=5
T=1s: cpu  user=1010 nice=10 system=205 idle=8080 iowait=55 irq=12 softirq=22 steal=5

差值：user=10 nice=0 system=5 idle=80 iowait=5 irq=2 softirq=2 steal=0
总差值 = 10+0+5+80+5+2+2+0 = 104

us = 10/104 × 100 ≈ 10%
sy = (5+2+2)/104 × 100 ≈ 9%   ← system + irq + softirq
id = 80/104 × 100 ≈ 77%
wa = 5/104 × 100 ≈ 5%
st = 0/104 × 100 = 0%
```


## 6. ★ 性能诊断阈值速查表

这是本文最重要的部分——将 vmstat 输出转化为可操作的诊断结论。

### 6.1 综合速查表

```
┌────────────────────────────────────────────────────────────────────┐
│                   ★ vmstat 性能诊断阈值速查表                       │
├──────────┬───────────────────┬─────────────────────────────────────┤
│ 指标      │ 阈值              │ 诊断结论                             │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ r         │ > CPU 核数        │ ★ CPU 过载，进程排队等 CPU           │
│           │ > 2× CPU 核数     │ ★★ 严重过载，响应时间急剧上升       │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ b         │ > 0 持续          │ ★ I/O 瓶颈，进程被 I/O 阻塞         │
│           │ > 5 持续          │ ★★ 严重 I/O 瓶颈                    │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ si        │ > 0 持续          │ ★ swap 活跃，内存不足               │
│ so        │ > 0 持续          │ ★ swap 活跃，内存不足               │
│           │ > 100 KB/s 持续   │ ★★ 严重内存压力，系统可能卡顿       │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ bi        │ 持续很高          │ 大量磁盘读取，检查缓存命中率         │
│ bo        │ 持续很高          │ 大量磁盘写入，检查写 I/O 来源        │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ in        │ > 50000/s         │ ★ 中断风暴（网络/磁盘密集）         │
│           │ > 100000/s        │ ★★ 极端中断负载                     │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ cs        │ > 10000/s         │ ★ 过度上下文切换，调度开销大        │
│           │ > 50000/s         │ ★★ 严重锁竞争或线程过多             │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ us        │ > 80%             │ CPU 密集于用户态计算                │
│ sy        │ > 30%             │ ★ 内核态开销大，检查系统调用频率     │
│           │ > 50%             │ ★★ 异常，可能有锁竞争或频繁系统调用  │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ id        │ < 10%             │ CPU 几乎没有空闲                    │
│           │ = 0%              │ ★★ CPU 完全饱和                     │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ wa        │ > 10%             │ ★ I/O 等待严重                      │
│           │ > 30%             │ ★★ I/O 是主要瓶颈                   │
├──────────┼───────────────────┼─────────────────────────────────────┤
│ st        │ > 5%              │ ★ 虚拟化环境，宿主机资源竞争        │
│           │ > 20%             │ ★★ 严重资源争抢，考虑迁移实例       │
└──────────┴───────────────────┴─────────────────────────────────────┘
```

### 6.2 组合诊断模式

单看一列往往不够，**组合多个列**才能精准定位瓶颈：

```
┌───────────────────────────────────────────────────────────────────┐
│                   组合诊断决策树                                   │
├───────────────────────────────────────────────────────────────────┤
│                                                                   │
│  r 高 + us 高 + id 低                                             │
│  └─► CPU 密集型应用（计算/编译/加密）                               │
│                                                                   │
│  r 高 + sy 高 + cs 高                                             │
│  └─► 锁竞争 / 过多线程 / 频繁系统调用                              │
│                                                                   │
│  b 高 + wa 高 + bo 高                                             │
│  └─► 磁盘写入瓶颈（数据库 WAL / 日志写入）                          │
│                                                                   │
│  b 高 + wa 高 + bi 高                                             │
│  └─► 磁盘读取瓶颈（缓存未命中 / 大数据扫描）                        │
│                                                                   │
│  si/so > 0 + free 低 + cache 低                                   │
│  └─► 物理内存不足，正在使用 swap                                   │
│                                                                   │
│  in 高 + us 不高 + sy 高                                          │
│  └─► 网络包风暴 / 磁盘中断过多                                     │
│                                                                   │
│  cs 高 + r 不高 + sy 高                                           │
│  └─► 进程频繁 sleep/wakeup（I/O 密集模型）                         │
│                                                                   │
│  wa 高 + b = 0                                                    │
│  └─► NFS/网络存储等待（不经过块设备层）                             │
│                                                                   │
│  st 高 + 所有其他指标正常                                          │
│  └─► 宿主机超卖，CPU 时间被偷                                     │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

### 6.3 真实案例分析

```
案例 1：CPU 过载
─────────────────
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
24  0      0 234567  12345 890123    0    0     2     5  890 1200 92  6  2  0  0

诊断：r=24 远超 8 核 CPU → CPU 严重过载
      us=92% → 用户态计算密集
      建议：优化应用算法，或增加 CPU 核心


案例 2：I/O 瓶颈
─────────────────
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 3  8      0 123456  34567 567890    0    0  5000  8000 2000 3000 10 15  5 70  0

诊断：b=8 → 大量进程阻塞在 I/O
      wa=70% → CPU 70% 时间在等 I/O
      bi=5000, bo=8000 → 读写都很密集
      建议：检查磁盘健康，考虑 SSD 升级


案例 3：内存不足
─────────────────
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2  3 2048000  12345   5678  34567  200  350   800  1200 1500 2500 40 20 10 30  0

诊断：swpd=2GB → 大量使用 swap
      si=200, so=350 → swap 活动频繁
      free=12MB → 物理内存几乎耗尽
      b=3 → 进程被 swap I/O 阻塞
      建议：增加物理内存，或减少应用内存使用
```

### 7.2 内核统计点的更新时机

```
┌──────────────────────────────────────────────────────────────┐
│  内核中各个计数器的更新时机                                    │
├──────────────────┬───────────────────────────────────────────┤
│ 计数器            │ 何时更新                                  │
├──────────────────┼───────────────────────────────────────────┤
│ procs_running    │ 进程加入/离开运行队列时（调度器）           │

```

## 8. 总结

### 8.1 vmstat 调用链

```
用户执行 "vmstat 1"
       │
       ▼
  解析参数（间隔=1s，次数=无限）
       │
       ▼
  打开 /proc/stat, /proc/vmstat, /proc/meminfo
       │
       ├──► 第 1 次读取：保存基线快照
       │         │
       │         ▼
       │    sleep(1)
       │         │
       │         ▼
       ├──► 第 2 次读取：计算差值 → 输出第 1 行
       │         │
       │         ▼
       │    sleep(1)
       │         │
       │         ▼
       ├──► 第 3 次读取：计算差值 → 输出第 2 行
       │         │
       │         ▼
       │       ...循环...
       │
       ▼
  退出
```
