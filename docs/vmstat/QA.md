## Q1: vmstat 首次输出的数值为什么和后续行差异很大？

**日期：** 2026-07-04  
**场景：** 执行 `vmstat 1` 后发现第一行的 cs、in、bi 等数值与后续行差异巨大（例如 cs=5 但第二行 cs=5000）  
**相关文件：** `docs/vmstat/reading/01_vmstat_architecture.md`（§5.3 首次输出的特殊性）  
**源码位置：** 内核 `fs/proc/stat.c:107-215` — `show_stat()`、`kernel/sched/cputime.c` — CPU 时间累计

### 回答

**★ 第一行是自系统启动以来的累计平均值，从第二行开始才是真实的区间增量数据。分析时应跳过第一行。**

#### 根本原因：/proc 文件中的累计计数器

vmstat 的所有速率数据来自 `/proc/stat` 和 `/proc/vmstat`，这些文件中的数值是**自启动以来的单调递增累计值**：

```c
// fs/proc/stat.c:196-206 — /proc/stat 输出 ctxt 累计值
seq_printf(p,
    "\nctxt %llu\n"           // ★ 自启动以来的上下文切换总次数
    "btime %llu\n"
    "processes %lu\n"
    "procs_running %lu\n"
    "procs_blocked %lu\n",
    nr_context_switches(),    // ★ 遍历所有 CPU 的 nr_switches 求和
    (unsigned long long)boottime.tv_sec,
    total_forks,
    nr_running(),
    nr_iowait());
```

```c
// fs/proc/stat.c:107-161 — CPU 时间也是累计值
// 遍历所有 CPU，累加自启动以来的全部时间
for_each_possible_cpu(i) {
    user   += cpustat[CPUTIME_USER];
    nice   += cpustat[CPUTIME_NICE];
    system += cpustat[CPUTIME_SYSTEM];
    idle   += get_idle_time(&kcpustat, i);
    // ... 全部是单调递增的累计值
}
```

#### 首次输出 vs 后续输出的计算差异

```
                  首次输出（第 1 行）              后续输出（第 2 行起）
                  ────────────────────           ────────────────────────
数据来源          /proc 文件的累计值               两次采样的差值
计算公式          累计值 / 系统运行时间             (值₂ - 值₁) / 采样间隔
含义              自启动以来的平均值               最近 N 秒的真实速率

示例（cs 列）：
  系统运行了 30 天，累计 ctxt = 12,960,000
  首次输出: cs = 12,960,000 / (30×86400) = 5/s  ← 30 天平均值！
  第二次采样: ctxt₁ = 12,960,000
  第三次采样: ctxt₂ = 12,965,000
  后续输出: cs = (12,965,000 - 12,960,000) / 1 = 5000/s  ← 真实速率
```

#### 不同列类型的行为差异

```
列类型          │ 首次输出行为                  │ 后续输出行为
───────────────┼──────────────────────────────┼──────────────────────────
瞬时值          │ 直接读取（无差异）            │ 直接读取（无差异）
(r, b, free,   │ ★ 这些列首次和后续            │
 buff, cache)  │   没有计算差异                │
───────────────┼──────────────────────────────┼──────────────────────────
累计计数器      │ 累计值 / uptime              │ 差值 / 采样间隔
(bi,bo,si,so,  │ ★ 差异巨大！                 │
 in, cs)       │ 长时间运行的系统上             │
               │ 首次值会被严重"稀释"           │
───────────────┼──────────────────────────────┼──────────────────────────
CPU 百分比      │ 累计 CPU 时间百分比           │ 区间 CPU 时间百分比
(us,sy,id,     │ ★ 长时间 idle 的系统上        │
 wa, st)       │ 首次 id 会接近 100%           │
```

#### 真实示例

```bash
$ vmstat 1 3
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 0  0      0 512340  34567 890123    0    0     5    10 1200    5  3  1 95  1  0  ← 第 1 行
 2  1 512340 510200  34567 890200    0    4    20    50 4500 6000 10  5 80  5  0  ← 第 2 行
 1  0 512340 511000  34567 890100    0    0     2     8 3800 5200  4  2 93  1  0  ← 第 3 行
```

```
★ 注意第 1 行的极端平均值：
  in=1200  → 启动以来总中断 / 总秒数 = 极低平均值
  cs=5     → 启动以来总切换 / 总秒数 = 极低平均值
  bi=5     → 启动以来总读入 / 总秒数 = 极低平均值

★ 第 2 行起才是真实数据：
  in=4500  → 实际每秒约 4500 次中断
  cs=6000  → 实际每秒约 6000 次上下文切换
  bi=20    → 实际每秒约 20 blocks 读入
```

#### 最佳实践

```bash
# ★ 方法 1：跳过第一行
vmstat 1 5 | tail -4    # 取后续 4 行有效数据

# ★ 方法 2：脚本中获取单次有效读数
vmstat 1 2 | tail -1    # 第二行才是真实数据

# ★ 方法 3：用 vmstat -w 加宽输出（更易读）
vmstat -w 1 3

# ★ 方法 4：如果只需要一次快照，用其他工具
sar 1 1                 # sar 没有"首次平均"问题
```

**一句话总结：** vmstat 第一行输出是 `累计值 / uptime` 得到的**自启动以来平均值**，运行时间越长，平均值越被"稀释"。从第二行起才是 `(采样₂ - 采样₁) / 间隔` 的**真实区间数据**。分析时永远跳过第一行。

## Q2: vmstat 的 cs 列（上下文切换）多少算正常？

**日期：** 2026-07-04  
**场景：** vmstat 输出中 cs 列数值波动较大，不知道什么范围算正常、什么范围需要关注  
**相关文件：** `docs/vmstat/reading/01_vmstat_architecture.md`（§2.5 system、§6.1 阈值速查表）  
**源码位置：** 内核 `kernel/sched/core.c:4501` — `rq->nr_switches++`、`kernel/sched/core.c:3820-3828` — `nr_context_switches()`

### 回答

**★ 没有绝对的"正常值"——cs 取决于工作负载类型。空闲系统 < 1000/s，一般服务器 1000-10000/s，高并发应用可达数万。关键是结合 r、sy 等列综合判断。**

#### cs 的内核计数机制

```c
// kernel/sched/core.c:4498-4528 — __schedule() 中的切换计数
if (likely(prev != next)) {
    rq->nr_switches++;     // ★ per-CPU 运行队列的累计切换计数器 +1
    RCU_INIT_POINTER(rq->curr, next);
    ++*switch_count;       // ★ 进程的 nvcsw 或 nivcsw +1

    trace_sched_switch(preempt, prev, next);
    rq = context_switch(rq, prev, next, &rf);  // 实际执行切换
}
```

```c
// kernel/sched/core.c:3820-3828 — 汇总所有 CPU 的切换次数
unsigned long long nr_context_switches(void)
{
    int i;
    unsigned long long sum = 0;

    for_each_possible_cpu(i)
        sum += cpu_rq(i)->nr_switches;  // ★ 遍历所有 CPU 的 nr_switches

    return sum;
}
```

```c
// fs/proc/stat.c:197 — 输出到 /proc/stat
"\nctxt %llu\n"    // ★ vmstat 读取此字段做差值
```

#### 两种上下文切换

```
__schedule() 中的 switch_count 指向：

自愿切换（voluntary，nvcsw++）：
  prev_state != 0 && !preempt
  → 进程主动让出 CPU
  → 场景：sleep()、等待 I/O、等待锁、epoll_wait()
  → switch_count = &prev->nvcsw

非自愿切换（involuntary，nivcsw++）：
  preempt = true（被更高优先级任务抢占）
  或时间片到期
  → switch_count = &prev->nivcsw

★ vmstat 的 cs 列 = nvcsw + nivcsw 的总和（不区分类型）
★ 可通过 pidstat -w 查看单进程的自愿/非自愿切换比例
```

#### cs 数值参考范围

```
┌──────────────────────────────────────────────────────────────────┐
│                   ★ cs 数值参考范围                                │
├────────────────┬────────────────┬────────────────────────────────┤
│ cs 范围         │ 系统状态       │ 典型场景                        │
├────────────────┼────────────────┼────────────────────────────────┤
│ < 1,000/s      │ ★ 非常空闲    │ 个人桌面空闲、新启动的服务器     │
│ 1,000 - 5,000  │ 正常           │ 一般 Web 服务器、数据库         │
│ 5,000 - 10,000 │ 中等负载       │ 高并发 Web 应用、编译任务       │
│ 10,000 - 50,000│ ★ 需要关注    │ 大量线程竞争、锁竞争激烈        │
│ > 50,000       │ ★★ 异常       │ 严重锁竞争、线程风暴            │
└────────────────┴────────────────┴────────────────────────────────┘

★ 注意：以上数值仅供参考，实际"正常值"取决于：
  - CPU 核心数（核心越多，自然切换越多）
  - 工作负载类型（I/O 密集型天然比 CPU 密集型切换多）
  - 线程数量（线程越多，切换越频繁）
```

#### 不同工作负载的 cs 特征

```
工作负载类型         │ 典型 cs/s   │ 原因
────────────────────┼────────────┼────────────────────────────────
空闲桌面             │ 100-500    │ 只有定时器和少量后台任务
Web 服务器（nginx）  │ 2,000-8,000│ 每个请求涉及 epoll 等待/唤醒
数据库（MySQL）      │ 3,000-15,000│ 大量线程池 + 锁竞争
编译任务（make -j）  │ 5,000-20,000│ 大量短生命周期进程 fork/exec
Java 应用（高并发）  │ 10,000-50,000│ GC 线程 + 工作线程竞争
锁竞争严重的程序     │ > 50,000   │ ★ futex 频繁 wait/wake 循环
```

#### 综合诊断：cs 不是孤立的指标

```
┌──────────────────────────────────────────────────────────────────┐
│                   ★ cs 的组合诊断                                  │
├────────────────────────────┬─────────────────────────────────────┤
│ 组合模式                    │ 诊断结论                             │
├────────────────────────────┼─────────────────────────────────────┤
│ cs 高 + r 高 + sy 高       │ ★ 线程过多 + 锁竞争                  │
│                            │ 建议：减少线程数，检查锁热点          │
├────────────────────────────┼─────────────────────────────────────┤
│ cs 高 + r 不高 + sy 高     │ ★ I/O 密集型（频繁 sleep/wakeup）    │
│                            │ 常见：网络/磁盘密集型应用            │
│                            │ 通常不需要优化                       │
├────────────────────────────┼─────────────────────────────────────┤
│ cs 高 + b 高 + wa 高       │ ★ I/O 瓶颈导致频繁阻塞/唤醒          │
│                            │ 建议：优化 I/O（SSD、缓存）          │
├────────────────────────────┼─────────────────────────────────────┤
│ cs 低 + r 高 + us 高       │ ★ CPU 密集型（很少切换）             │
│                            │ 进程一直占着 CPU 不释放              │
├────────────────────────────┼─────────────────────────────────────┤
│ cs 低 + id 高              │ ★ 系统空闲                           │
│                            │ 正常状态                            │
└────────────────────────────┴─────────────────────────────────────┘
```

#### 进一步定位高 cs 的来源

```bash
# 1. 查看哪些进程的上下文切换最多
pidstat -w -p ALL 1 5
# 输出 nvcsw/s（自愿切换）和 nivcsw/s（非自愿切换）

# 2. 区分自愿 vs 非自愿切换
# nvcsw 高 → 进程频繁等待（I/O、锁、事件）→ 通常正常
# nivcsw 高 → 进程被频繁抢占 → 可能线程过多

# 3. 用 perf 追踪调度热点
perf record -e sched:sched_switch -a -- sleep 5
perf report
# 找到切换最频繁的进程和调用栈

# 4. 用 bpftrace 追踪切换原因
bpftrace -e '
  tracepoint:sched:sched_switch {
    @[comm] = count();
  }
'
# ★ 找出切换次数最多的进程名
```

**一句话总结：** cs 没有绝对的"正常值"——空闲系统 < 1000/s，一般服务器 1000-10000/s，高并发应用可达数万。关键看**组合模式**：`cs 高 + r 高 + sy 高` = 锁竞争/线程过多需要优化；`cs 高 + r 不高` = I/O 密集型的正常现象。用 `pidstat -w` 定位哪些进程的切换最多。
