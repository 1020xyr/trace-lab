## 一、三种基准测试场景

### 1.1 ★ 延迟测试：iodepth=1, numjobs=1, direct=1

```bash
# ★ 延迟测试的黄金配置
fio --name=latency_test \
    --ioengine=libaio \
    --direct=1 \
    --rw=randwrite \
    --bs=4k \
    --iodepth=1 \
    --numjobs=1 \
    --filename=/dev/vdb \
    --runtime=30 \
    --time_based \
    --lat_percentiles=1
```

**为什么每个参数都不可少：**

| 参数 | 值 | 原因 | 设错的后果 |
|------|-----|------|----------|
| `iodepth` | ★ 1 | 排除排队效应，测的是单个 I/O 的真实延迟 | iodepth > 1 会引入排队延迟，测的不是设备延迟 |
| `numjobs` | 1 | 排除多 job 竞争 | numjobs > 1 会引入锁竞争和调度开销 |
| `direct` | ★ 1 | 绕过 page cache，测的是设备延迟 | direct=0 命中 page cache 时延迟 ~0.5μs，不真实 |
| `bs` | 4k | 最小常用块大小，最能暴露延迟问题 | bs 越大，延迟越高（传输时间更长），掩盖真实延迟 |
| `rw` | randwrite | 随机写最考验设备延迟 | 顺序写可能触发设备预取，延迟偏低 |

**iodepth 对延迟的影响（图解）：**

```
iodepth=1（正确测延迟）：
  I/O 1: [──── 120μs ────]
  I/O 2:                  [──── 115μs ────]
  I/O 3:                                  [──── 125μs ────]
  ★ 每个 I/O 独占设备，延迟 = 设备真实延迟

iodepth=32（延迟被排队放大）：
  I/O 1-32: 同时提交
            [─── I/O 1: 120μs ───]
            [─── I/O 2: 135μs ────]  ← 等待 I/O 1 让出设备资源
            [──── I/O 3: 180μs ──────]  ← 排队更久
            ...
            [──────── I/O 32: 800μs ────────]  ← 排队延迟！
  ★ 平均延迟 = ~450μs（不是设备真实延迟）

结论：iodepth=1 测的是"设备延迟"，iodepth=32 测的是"排队延迟"
```

### 1.2 ★ 吞吐测试：iodepth=32, numjobs=4, direct=1

```bash
# ★ 吞吐测试（最大化带宽）
fio --name=throughput_test \
    --ioengine=libaio \
    --direct=1 \
    --rw=read \
    --bs=1m \
    --iodepth=32 \
    --numjobs=4 \
    --filename=/dev/vdb \
    --runtime=30 \
    --time_based \
    --group_reporting
```

| 参数 | 值 | 原因 |
|------|-----|------|
| `iodepth` | ★ 32 | 充分利用设备队列深度 |
| `numjobs` | 4 | 多 job 并行，充分利用多队列 |
| `bs` | ★ 1m | 大块 I/O 减少 per-IO 开销，最大化吞吐 |
| `rw` | read | 顺序读测最大读带宽 |

**iodepth 与吞吐的关系：**

```
吞吐 (MB/s)
  │
2000│                    ★ 拐点：iodepth=32
    │                  ╱───────────────────
1500│                ╱
    │              ╱
1000│            ╱
    │          ╱
 500│        ╱
    │      ╱
    │    ╱
  0 │──╱──────────────────────────────
    1    4    8   16   32   64  128  iodepth
    
    ★ 拐点之后继续增大 iodepth：
      吞吐不再增长（设备已满）
      延迟持续增加（排队更长）
```

### 1.3 ★ IOPS 测试：bs=4k, iodepth=128

```bash
# ★ IOPS 测试（最大化每秒 I/O 次数）
fio --name=iops_test \
    --ioengine=libaio \
    --direct=1 \
    --rw=randread \
    --bs=4k \
    --iodepth=128 \
    --numjobs=4 \
    --filename=/dev/vdb \
    --runtime=30 \
    --time_based \
    --group_reporting
```

| 参数 | 值 | 原因 |
|------|-----|------|
| `bs` | ★ 4k | 小 I/O 最大化 IOPS（每次传输量最小） |
| `iodepth` | ★ 128 | 高并发保持设备队列满 |
| `rw` | randread | 随机读是 IOPS 的标准测试模式 |
| `numjobs` | 4 | 多 job 并行利用多队列 |

**IOPS 与 bs 的关系：**

```
IOPS
  │
500K│ ★ bs=4k
    │ ████████████
400K│ ████████████
    │ ████████████
300K│ ████████████
    │ ████████████    bs=64k
200K│ ████████████    ████████
    │ ████████████    ████████   bs=1m
100K│ ████████████    ████████   ████
    │ ████████████    ████████   ████
  0 │████████████████████████████████
    4k    8k   16k   64k   256k   1m   bs
    
    ★ IOPS × bs = 带宽
      bs=4k, IOPS=500K → 带宽 = 2 GB/s
      bs=1m, IOPS=2K → 带宽 = 2 GB/s（同一设备）
```

## 二、fio 输出与 blktrace/btt 的精确对照

### 2.1 时间线对照图

```
fio 用户态视角：

  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ get_io_u │         │ io_submit│         │getevents │
  │ (准备)   │         │ (提交)   │         │ (收割)   │
  └────┬─────┘         └────┬─────┘         └────┬─────┘
       │                    │                    │
       │  start_time        │  issue_time        │  complete_time
       │                    │                    │
       │←──── slat ────→│←────── clat ──────→│
       │←────────────── lat ─────────────────→│


blktrace 内核视角：

  ┌──────────┐         ┌──────────┐         ┌──────────┐
  │ Q (Queue)│         │ D (Issue)│         │C(Complete)│
  │ bio 入队 │         │ 下发设备 │         │ 中断完成  │
  └────┬─────┘         └────┬─────┘         └────┬─────┘
       │                    │                    │
       │←──── Q2D ─────→│←───── D2C ──────→│
       │←────────────── Q2C ───────────────→│


关键时间点对应关系：
  fio start_time   →  大约在 Q 之前（用户态准备）
  fio issue_time   →  大约在 Q 之后、D 之前（io_submit 返回）
  fio complete_time →  大约在 C 之后（getevents 返回）
```

### 2.2 精确对照表

```
┌───────────┬──────────────┬─────────────────────┬─────────────────────────────────┐
│ fio 指标   │ blktrace 对应 │ 是否精确等价         │ 差异说明                         │
├───────────┼──────────────┼─────────────────────┼─────────────────────────────────┤
│ slat       │ Q2D          │ ★ 近似，不精确       │ fio slat = 用户态准备+提交       │
│            │              │                     │ blktrace Q2D = 内核 bio 入队+调度│
│            │              │                     │ fio slat 包含系统调用开销         │
│            │              │                     │ Q2D 包含内核调度开销             │
├───────────┼──────────────┼─────────────────────┼─────────────────────────────────┤
│ clat       │ D2C          │ ★ 近似，不精确       │ fio clat = io_submit→getevents  │
│            │              │                     │ 包含完成通知延迟（中断→用户态）  │
│            │              │                     │ D2C = 设备下发→中断完成          │
│            │              │                     │ 不包含用户态通知延迟             │
├───────────┼──────────────┼─────────────────────┼─────────────────────────────────┤
│ lat        │ Q2C          │ ★★ 最接近           │ 两者都是端到端延迟               │
│            │              │                     │ fio lat 略大于 Q2C              │
│            │              │                     │ （包含用户态首尾开销）           │
└───────────┴──────────────┴─────────────────────┴─────────────────────────────────┘
```

### 2.3 差异的根因分析

```
fio slat vs blktrace Q2D：

  fio:  start_time ──→ issue_time
        │ get_io_u()   │ io_submit() 返回
        │ 用户态准备    │
        │ fill_io_u()  │ 包含 io_submit 系统调用开销
        │              │
  blktrace: Q ──→ D
              │ bio 入队 │ 下发到设备驱动
              │ 内核内部  │ 包含 I/O 调度器处理
              │          │

  ★ fio slat 通常 < Q2D（slat 不含内核调度，Q2D 含）
    但也有 slat > Q2D 的情况（io_submit 系统调用慢时）

fio clat vs blktrace D2C：

  fio:  issue_time ──→ complete_time
        │ io_submit 后 │ getevents 返回
        │ 包含等待完成  │ 包含中断处理
        │ 通知的时间    │ 和用户态收割时间
        
  blktrace: D ──→ C
              │ 设备驱动 │ 中断完成
              │ 纯设备    │ 纯硬件
              │ 处理时间  │

  ★ fio clat 通常 > D2C（clat 包含完成通知延迟）
```

## 三、fio + blktrace 联合分析流程

### 3.1 联合分析场景：定位延迟瓶颈

```bash
# ★ 场景：fio 测出延迟异常，需要定位瓶颈在哪一层

# 第 1 步：同时运行 fio + blktrace
blktrace -d /dev/vdb -o latency_diag -w 60 &
BLKTRACE_PID=$!
sleep 1

fio --name=diag_test \
    --ioengine=libaio \
    --direct=1 \
    --rw=randwrite \
    --bs=4k \
    --iodepth=32 \
    --numjobs=1 \
    --filename=/dev/vdb \
    --runtime=30 \
    --time_based \
    --write_lat_log=/tmp/fio_lat \
    --log_avg_msec=100

wait $BLKTRACE_PID

# 第 2 步：分析 fio 输出
# 关注：slat avg vs clat avg
# 如果 slat >> clat → 瓶颈在软件层（用户态提交慢）
# 如果 clat >> slat → 瓶颈在设备层（设备处理慢）

# 第 3 步：分析 blktrace/btt
blkparse -i latency_diag -o /tmp/blktrace_parsed.txt
btt -i latency_diag.bin -o /tmp/btt_output

# 第 4 步：对比 fio 和 btt 的延迟指标
```

### 3.2 瓶颈定位决策树

```
fio 测出延迟异常（P99 > 预期）
         │
         ▼
  ┌── 看 fio 输出的 slat 和 clat ──┐
  │                                 │
  │  slat 异常高？    clat 异常高？ │
  │  (avg > 10μs)    (avg > 预期)  │
  │                                 │
  └────────┬────────────┬──────────┘
           │            │
     ┌─────┴─────┐     │
     │           │     │
     ▼           ▼     ▼
   slat 高    clat 高   两者都高
     │           │     │
     ▼           ▼     ▼
  软件层      设备层   全链路瓶颈
  瓶颈        瓶颈
     │           │     │
     ▼           ▼     ▼
  排查方向：   排查方向：  两边都要排查
  · ioengine   · 设备健康
  · 系统调用   · 队列深度
    开销       · 设备 GC
  · CPU 瓶颈   · 驱动问题
  · 内核版本
```

### 3.3 实战对照示例

```
fio 输出：
  write: IOPS=4800
  slat (nsec): avg=2100.00       ← 2.1μs
  clat (usec): avg=180.00        ← 180μs
  lat  (usec): avg=182.10        ← 182μs

btt 输出：
  ALL        Q2D       D2C       Q2C
  avg     5.2μs    175.3μs    180.5μs

对照分析：
┌──────────────┬──────────┬──────────┬──────────────────────────────────┐
│ 指标          │ fio      │ btt      │ 分析                              │
├──────────────┼──────────┼──────────┼──────────────────────────────────┤
│ 提交延迟      │ 2.1μs    │ 5.2μs    │ btt Q2D > fio slat               │
│              │ (slat)   │ (Q2D)    │ ★ Q2D 含内核调度延迟 3.1μs       │
│              │          │          │ → 内核 I/O 调度有 ~3μs 开销       │
├──────────────┼──────────┼──────────┼──────────────────────────────────┤
│ 设备延迟      │ 180μs    │ 175.3μs  │ fio clat > btt D2C               │
│              │ (clat)   │ (D2C)    │ ★ clat 含完成通知延迟 ~5μs       │
│              │          │          │ → 中断 → 用户态通知有 ~5μs 开销   │
├──────────────┼──────────┼──────────┼──────────────────────────────────┤
│ 端到端延迟    │ 182μs    │ 180.5μs  │ ★ 两者接近（差异 < 1%）          │
│              │ (lat)    │ (Q2C)    │ → 端到端测量一致性好              │
└──────────────┴──────────┴──────────┴──────────────────────────────────┘

★ 结论：
  1. 设备延迟（D2C/clat）= 175-180μs → 正常 NVMe 范围 ✓
  2. 内核调度开销（Q2D - slat）= 3.1μs → 正常 ✓
  3. 完成通知开销（clat - D2C）= 4.7μs → 正常 ✓
  4. 如果需要降低延迟，瓶颈在设备层（占 96%）
```

### 3.4 延迟异常的常见根因

```
异常模式                          │ 根因                        │ 解决方案
─────────────────────────────────┼────────────────────────────┼─────────────────
slat 高（> 50μs）                │ io_submit 系统调用慢         │ 升级内核
                                 │ CPU 瓶颈（usr% 高）         │ 减少 numjobs
─────────────────────────────────┼────────────────────────────┼─────────────────
clat 高（> 设备标称值）          │ 设备队列太深                 │ 降低 iodepth
                                 │ 设备 GC 干扰                │ 预格式化/trim
                                 │ 驱动 bug                    │ 升级驱动
─────────────────────────────────┼────────────────────────────┼─────────────────
Q2D 高（btt 显示 > 50μs）       │ I/O 调度器开销              │ 换 none/mq-deadline
                                 │ 多路径/DM 层开销            │ 绕过 DM 直测
─────────────────────────────────┼────────────────────────────┼─────────────────
D2C 抖动大（P99 >> P50）         │ 设备内部 GC                 │ 定期 fio trim
                                 │ 热节流                      │ 改善散热
                                 │ 其他进程 I/O 干扰           │ 独占设备测试
─────────────────────────────────┼────────────────────────────┼─────────────────
lat 正常但 IOPS 低               │ iodepth 不够                 │ 增大 iodepth
                                 │ numjobs 不够                 │ 增大 numjobs
                                 │ CPU 瓶颈                    │ 绑核/增加核数
```

## 四、常见错误配置和误导性结论

### 4.2 正确的对比测试方法

```bash
# ★ A/B 对比测试的正确方法：
# 1. 除被测变量外，其他参数完全一致
# 2. 在独占设备上测试
# 3. 运行足够长的时间（≥ 30 秒）
# 4. 重复 3 次取平均值
# 5. 报告标准差（stdev）

# 示例：对比两种 I/O 调度器的性能
# 先设置调度器
echo mq-deadline > /sys/block/vdb/queue/scheduler

# 测试 1
fio --name=test1 --ioengine=libaio --direct=1 --rw=randwrite \
    --bs=4k --iodepth=32 --filename=/dev/vdb --runtime=30 --time_based

# 切换调度器
echo none > /sys/block/vdb/queue/scheduler

# 测试 2（★ 其他参数完全一样）
fio --name=test2 --ioengine=libaio --direct=1 --rw=randwrite \
    --bs=4k --iodepth=32 --filename=/dev/vdb --runtime=30 --time_based
```
