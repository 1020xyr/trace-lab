## Q1: iostat %util=100% 一定是瓶颈吗？

**日期：** 2026-07-04
**场景：** 性能诊断时观察到 %util 持续 100%，但不确定是否真的是 I/O 瓶颈
**相关文件：** reading/01_iostat_architecture.md（%util 计算逻辑）

### 回答

**不一定。** %util=100% 的含义是"整个采样期间设备一直有 I/O 在处理"，但**不等于设备无法处理更多 I/O**。

#### %util 的计算本质

```
%util = Δio_time / (Δt × 1000) × 100%

io_time（/proc/diskstats 字段 13）的递增条件：
  "只要有至少 1 个 I/O 在处理中，每个 jiffy（1ms）就 +1"

所以 %util=100% 只说明：
  "设备在整个 1 秒采样期内，始终有 ≥1 个 I/O 在忙"
```

#### 关键区分：单队列 vs 多队列

```
┌─────────────────────────────────────────────────────────────┐
│ 单队列设备（HDD、virtio-blk 默认模式）                      │
│                                                              │
│   %util=100% ≈ 设备饱和                                     │
│   原因：同一时刻只能处理 1 个 I/O                            │
│         100% util = 设备一直在忙 = 没有空闲时间               │
│                                                              │
│   结论：★ 对于单队列设备，%util=100% 确实意味着饱和          │
├─────────────────────────────────────────────────────────────┤
│ 多队列设备（NVMe SSD、配置了多队列的 virtio-blk）           │
│                                                              │
│   %util=100% ≠ 设备到极限                                   │
│   原因：多个队列可以并行处理 I/O                             │
│         100% util 只代表"至少一个队列一直有活干"             │
│         其他队列可能还有大量余量                             │
│                                                              │
│   结论：★ 对于多队列设备，%util=100% 需要结合 await 判断     │
│          - await 低 → 设备还有处理能力，不是瓶颈             │
│          - await 高 → 真的过载了                             │
└─────────────────────────────────────────────────────────────┘
```

#### 诊断思维链

```
%util = 100% 时：
│
├── await < 5ms（SSD 正常范围）
│   → 设备忙但响应快，★ 不一定有问题
│   → 检查 avgqu-sz 是否稳定
│   → 检查应用延迟是否受影响
│
├── await 5~20ms（需关注）
│   → 设备接近饱和，开始有排队
│   → 用 blktrace 分析 Q2D vs D2C
│
└── await > 20ms（明确异常）
    → ★ 设备过载，I/O 是瓶颈
    → 找产生 I/O 最多的进程：pidstat -d 2 3
    → 深入分析：blktrace + btt
```

#### 实际验证方法

```bash
# 1. 看 await 配合 %util
iostat -x -y 1 5
# 如果 %util=100% 但 await=1ms → SSD 正常满载

# 2. 看队列深度
# avgqu-sz 稳定在合理范围 → 设备可以承受
# avgqu-sz 持续增长 → 即将过载

# 3. 看应用层延迟
# 用 blktrace + btt 看 D2C（设备实际处理延迟）
# D2C < 1ms → SSD 硬件正常，是 IOPS 总量高
```

#### ★ 结论

**%util=100% 不是"I/O 瓶颈"的充分条件。** 必须结合 await 和 avgqu-sz 综合判断：
- %util=100% + await 低 + avgqu-sz 稳定 → 设备忙但能扛住（SSD 常见）
- %util=100% + await 高 + avgqu-sz 增长 → 设备过载，确实是瓶颈

## Q2: await 高但 %util 低怎么解释？

**日期：** 2026-07-04
**场景：** iostat 显示 await=30ms 但 %util 只有 15%，看起来很矛盾
**相关文件：** reading/01_iostat_architecture.md（await 和 %util 的计算逻辑）

### 回答

**await 高但 %util 低 = 设备不忙但偶尔的 I/O 很慢。** 这并不矛盾，两者衡量的是不同维度。

#### 两者衡量不同维度

```
%util → 时间维度："设备有多少时间在忙"
await → 延迟维度："每个 I/O 平均花了多久"

类比：
  %util = "收银台有多少时间在工作"
  await = "每个顾客平均等了多久"

  1 小时内只有 5 个顾客，但每个顾客都聊了 10 分钟：
    %util = 50min/60min = 83%（比较忙）
    await = 10min（等很久）

  1 小时内只有 2 个顾客，但每个都花了 10 分钟：
    %util = 20min/60min = 33%（不太忙）
    await = 10min（还是等很久）
```

#### 常见原因

```
原因 1：大请求（最常见）
  rareq-sz 或 wareq-sz 很大（如 128KB、256KB）
  大请求的绝对延迟本来就高
  但 IOPS 很低（r/s 或 w/s 只有几十），所以设备大部分时间空闲

  例：128KB 顺序读
    r/s = 50, rareq-sz = 128KB
    await = 10ms（128KB 数据读取需要时间）
    %util = 50 × 10ms / 1000ms = 50%

  这是正常行为，不是瓶颈。

原因 2：偶发慢 I/O
  设备大部分时间空闲，但偶尔出现慢 I/O：
  - SSD GC（垃圾回收）导致的延迟尖峰
  - HDD 磁头寻道（随机读位置跨度大）
  - 设备温度过高导致降频

  await 被少量慢 I/O 拉高（平均值敏感）
  但 %util 低说明不是持续问题。

原因 3：I/O 调度延迟
  I/O 请求在队列中等待调度（Q2D 高），但设备处理很快（D2C 低）
  此时设备利用率不高，但端到端延迟大

  可能原因：cgroup I/O 限制、I/O 优先级设置
```

#### 诊断方法

```bash
# 1. 看请求大小
iostat -x -y 1 3
# 如果 rareq-sz > 64KB → 大请求导致的高 await，正常

# 2. 看 IOPS
# r/s 或 w/s 很低（< 100）+ await 高 → 少量慢 I/O
# 不是持续压力，而是偶发事件

# 3. 用 blktrace 看延迟分布
blktrace -d /dev/vdb -o trace -w 10
btt -i trace.bin -o btt_out
# 看 Q2C 直方图 → 是均匀分布还是有少量异常值？
# 如果有少量 >100ms 的 I/O 拉高了平均值 → 偶发问题

# 4. 检查设备健康
smartctl -a /dev/sda  # SSD/HDD 健康状态
dmesg | grep -i error # 内核错误日志
```

#### 对比表

```
场景                    │ %util │ await │ r/s   │ 是否瓶颈 │ 处理方式
───────────────────────┼───────┼───────┼───────┼─────────┼────────────
大量 4K 随机写          │ ~100% │ 2ms   │ 20000 │ 看场景   │ SSD 正常
128K 顺序读（少量）     │ 15%   │ 10ms  │ 50    │ ★ 不是   │ 无需处理
偶发 SSD GC             │ 5%    │ 50ms  │ 10    │ 偶发     │ 监控趋势
队列堆积                │ 30%   │ 30ms  │ 100   │ 可能是   │ 检查 Q2D
```

#### ★ 结论

**await 高 + %util 低 = 少量慢 I/O，通常不是瓶颈。** 先检查请求大小（rareq-sz/wareq-sz），如果是大请求则完全正常。如果请求小但 await 高，用 blktrace 分析是 Q2D 高还是 D2C 高，定位具体原因。

## Q3: iostat 首次输出和后续输出为什么不同？

**日期：** 2026-07-04
**场景：** 运行 `iostat -x 1` 时发现第一次输出的数值和后续输出差异很大，不确定哪个是真实的
**相关文件：** `reading/01_iostat_architecture.md`、`reading/02_command_reference.md`

### 回答

**★ 首次输出是"开机以来的累积平均值"，后续输出才是"采样间隔内的增量速率"。** 这是 iostat 最容易让人困惑的设计，也是实践中最常踩的坑。

#### 两次输出的本质差异

```
┌─────────────────────────────────────────────────────────────────┐
│                    iostat 1 3                                    │
│                    │    │    │    │                              │
│                    t0   t1   t2   t3                            │
│                    ▼    ▼    ▼    ▼                              │
│                  read0 read1 read2 read3                         │
│                                                                  │
│  首次输出 = (read0 - boot_value) / (t0 - boot_time)             │
│             ↑                                                     │
│             开机以来的平均值（可能跨越数天！）                      │
│                                                                  │
│  第 2 次 = (read1 - read0) / 1s    ← 真正的 1 秒内速率           │
│  第 3 次 = (read2 - read1) / 1s    ← 真正的 1 秒内速率           │
│                                                                  │
│  ★ -y 参数跳过首次输出，直接从第 2 次开始                         │
│     iostat -y 1 3 → 只有 3 次间隔采样输出                        │
└─────────────────────────────────────────────────────────────────┘
```

（来源：reading/01_iostat_architecture.md 第 192-199 行）

#### 数据源：/proc/diskstats 的累积计数器

```
/proc/diskstats 中的所有字段都是"开机以来的累积值"：

  vdb: reads=211572  writes=5405125  sectors_read=13505762  io_time=635086
       ↑               ↑               ↑                     ↑
       开机以来总读取  开机以来总写入    开机以来总扇区         开机以来总忙时间

iostat 的计算方式：
  首次输出：当前值 / 开机时长 = 开机以来的平均速率
  后续输出：(当前值 - 上次值) / 间隔 = 区间内的真实速率

★ 如果机器运行了很久（几天、几周），首次输出是长时间平均值，
  完全不能反映当前的 I/O 状态！
```

#### ★ 具体例子：首次输出 vs 后续输出的差异

```
假设机器运行了 7 天（604800 秒），当前突然有大量 I/O：

/proc/diskstats 在 t=0 时刻：
  reads_completed = 1,000,000（7 天累积）
  io_time = 500,000ms（7 天累积）

/proc/diskstats 在 t=1 时刻（1 秒后）：
  reads_completed = 1,010,000（1 秒内增加了 10,000！）
  io_time = 500,950ms（1 秒内增加了 950ms）

首次输出：
  r/s = 1,000,000 / 604800 ≈ 1.65     ← 几乎无 I/O
  %util = 500,000 / (604800 × 1000) ≈ 0.08%  ← 看起来空闲

第 2 次输出：
  r/s = (1,010,000 - 1,000,000) / 1 = 10,000  ← ★ 真实的 I/O 速率！
  %util = (950 / 1000) × 100 = 95%             ← ★ 设备几乎满载！

★ 首次输出完全掩盖了当前的 I/O 风暴！
```

#### -y 参数的作用

```bash
# 不加 -y（默认）：首次输出 = 开机以来平均值（误导！）
iostat -x 1 5
# 第 1 次：开机以来平均 → 可能很低
# 第 2-5 次：每秒真实速率 → 正确

# 加 -y：跳过首次输出，全部都是区间增量（★ 推荐）
iostat -x -y 1 5
# 第 1-5 次：每秒真实速率 → 全部正确
```

（来源：reading/02_command_reference.md 第 78、155-168 行）

#### ★ 实践规则

```
┌──────────────────────────────────────────────────────────────┐
│ ★ 黄金法则：间隔采样时永远加 -y                               │
│                                                              │
│ 正确用法：                                                    │
│   iostat -x -y 1 10          # 持续监控                      │
│   iostat -x -y -t 1 > log    # 记录日志（带时间戳）           │
│   iostat -x -y vdb 1 5       # 只看某个设备                   │
│                                                              │
│ 错误用法（首次输出会误导）：                                   │
│   iostat -x 1 10             # ✗ 首次输出无意义               │
│   iostat -x 1 > log          # ✗ 日志第一条是错误数据         │
│                                                              │
│ 唯一例外：                                                    │
│   iostat -x                  # 不带间隔，只看一次累积统计      │
│   → 这是有意查看开机以来的总概况，此时不加 -y                   │
└──────────────────────────────────────────────────────────────┘
```

#### 验证：首次输出确实是 since-boot 平均值

```bash
# 方法 1：对比 uptime 和首次输出的数值
uptime  # 查看开机时长
iostat -x 1 2
# 首次输出的 r/s = 总读取次数 / 开机秒数
# 可以手动验证：
cat /proc/diskstats | grep vdb  # 读取累积值
# 用 累积读取数 / 开机秒数 ≈ 首次输出的 r/s

# 方法 2：刚重启后立即运行（首次输出 ≈ 后续输出）
# 因为开机时间短，平均值接近当前值

# 方法 3：运行很久后再运行（首次输出 ≪ 后续输出）
# 开机时间很长，平均值被大量历史数据稀释
```

**一句话总结：** `iostat` 首次输出是开机以来的累积平均值（可能跨越数天），完全不能反映当前 I/O 状态。后续输出才是区间增量速率。**间隔采样时永远加 `-y`**（`iostat -x -y 1`），这是 iostat 使用中最重要的实践规则。
