## 1. iostat 是什么

**iostat**（Input/Output Statistics）是 sysstat 包中的 I/O 统计工具，通过读取 `/proc/diskstats` 和 `/proc/stat` 计算 CPU 利用率和块设备 I/O 统计。

```
┌────────────────────────────────────────────────────┐
│                   iostat                           │
│  读取 /proc/stat      → CPU 利用率（%user, %iowait）│
│  读取 /proc/diskstats → 设备 I/O 统计（tps, kB/s） │
├────────────────────────────────────────────────────┤
│                   内核层                            │
│  /proc/stat:      cpu 时间累计（jiffies）           │
│  /proc/diskstats: 每个块设备的累计 I/O 计数器       │
├────────────────────────────────────────────────────┤
│                   Block Layer                      │
│  每次 I/O 完成 → 更新 disk_stats 计数器            │
│  每次 context switch → 更新 cpu 时间计数器          │
└────────────────────────────────────────────────────┘
```

### 核心特点

| 特点 | 说明 |
|------|------|
| **宏观视角** | 设备级聚合统计，看不到单个 I/O |
| **低开销** | 只读取两个 proc 文件，几乎无性能影响 |
| **间隔采样** | 两次读取的差值 = 该时段内的 I/O 速率 |
| **适合监控** | 快速判断"I/O 是不是瓶颈" |

## 2. iostat 与 blktrace 的关系

```
iostat                              blktrace
──────                              ────────
宏观统计                              微观追踪
设备级聚合                            单个 I/O 级
读取 /proc/diskstats                  通过 tracepoint 采集
无额外开销                            有一定开销（relay buffer）
适合"有没有问题"                       适合"问题出在哪"
看到 %util=100%                      看到每个 I/O 的 Q→D→C
看到 await=5ms                       看到 D2C=147μs
```

**诊断流程：**
```
1. iostat -x 1 → 发现 %util 高 / await 高（宏观异常）
                    ↓
2. blktrace + blkparse + btt → 追踪每个 I/O 的延迟细节（微观定位）
                    ↓
3. btt 延迟分析 → 是 Q2D 高（软件瓶颈）还是 D2C 高（硬件瓶颈）
```

**类比：**
- iostat ≈ 交通摄像头（统计每分钟经过多少车、平均速度）
- blktrace ≈ GPS 追踪器（每辆车的完整行驶轨迹）

**iostat 怎么用这些数据：**

```
iostat 间隔采样原理：

t=0s 读取 /proc/diskstats → reads_0=199296, writes_0=5170714
t=1s 读取 /proc/diskstats → reads_1=199351, writes_1=5186714

r/s = (reads_1 - reads_0) / (t1 - t0) = (199351 - 199296) / 1 = 55
w/s = (writes_1 - writes_0) / 1 = 16000
```

**思考题：**
1. `/proc/diskstats` 的数据是累计值还是速率？（答：累计值，iostat 自己算差值）
2. 为什么第一次 iostat 输出是从开机到现在的平均值？（答：只有一个采样点，只能算开机以来的平均）
3. `-y` 参数的作用？（答：跳过第一次累计统计，只显示间隔采样数据）

### Step 2: 基础输出 — CPU + 设备统计

**目标：** 理解 iostat 默认输出中每个字段的含义

**命令：** `iostat`

**输出字段：**

```
avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           1.11    0.01    0.48    0.03    0.00   98.37

Device             tps    kB_read/s    kB_wrtn/s    kB_dscd/s    kB_read    kB_wrtn    kB_dscd
vda               2.45         7.96       154.87         0.00   13779120  268092872          0
vdb               2.94         3.12        13.13         0.00    5393805   22723068          0
```

| 字段 | 含义 | 计算方式 |
|------|------|---------|
| `%user` | 用户态 CPU 时间占比 | user_jiffies / total_jiffies |
| `%iowait` | CPU 等待 I/O 完成的时间占比 | iowait_jiffies / total_jiffies |
| `tps` | 每秒传输数（IOPS） | (reads + writes) / interval |
| `kB_read/s` | 每秒读吞吐量 | read_sectors × 512 / 1024 / interval |
| `kB_wrtn/s` | 每秒写吞吐量 | write_sectors × 512 / 1024 / interval |

**思考题：**
1. `%iowait` 高是否意味着 I/O 是瓶颈？（答：不一定，%iowait 高也可能只是 CPU 空闲且恰好有 I/O 在等待）
2. `tps` 和 IOPS 是同一个概念吗？（答：基本等价，但 tps 可能包含合并后的传输）

### Step 3: 扩展输出 — iostat -x 的每个字段

**目标：** 理解扩展输出中每个指标的诊断价值

**命令：** `iostat -x`

**核心指标详解：**

| 指标 | 含义 | 正常范围 | 异常信号 |
|------|------|---------|---------|
| `r/s` | 每秒读 I/O 数 | 取决于工作负载 | 突增可能有预读或扫描 |
| `w/s` | 每秒写 I/O 数 | 取决于工作负载 | 突增可能有写回风暴 |
| `rkB/s` | 每秒读吞吐 | 取决于设备能力 | 接近设备上限=饱和 |
| `wkB/s` | 每秒写吞吐 | 取决于设备能力 | 接近设备上限=饱和 |
| `r_await` | 读平均延迟(ms) | < 5ms (SSD) | > 20ms 需要排查 |
| `w_await` | 写平均延迟(ms) | < 5ms (SSD) | > 20ms 需要排查 |
| `await` | 总平均延迟(ms) | < 5ms (SSD) | 应用感知延迟 |
| `r_rqm/s` | 读合并率 | 顺序读时高 | 0 表示完全随机 |
| `w_rqm/s` | 写合并率 | 顺序写时高 | 0 表示完全随机 |
| `avgqu-sz` / `aqu-sz` | 平均队列深度 | < 设备队列深度 | 过大=排队严重 |
| `%util` | 设备利用率 | < 80% | 100% = 设备饱和 |

**思考题：**
1. `%util=100%` 是否意味着设备已到极限？（答：对于单队列设备是的，但对于多队列 SSD 不一定）
2. `await` 包含哪些延迟？（答：排队延迟 + 设备处理延迟 = Q2C）

### Step 4: 间隔采样模式 — 实时观测

**目标：** 理解间隔采样的原理和 `-y` 参数的作用

```bash
# 每 1 秒采样一次，共 10 次
iostat -x 1 10

# 跳过首次累计统计（推荐！）
iostat -x -y 1 10

# 持续采样（不指定次数）
iostat -x 1
```

**间隔采样原理：**
```
t=0    t=1    t=2    t=3
 │      │      │      │
 ▼      ▼      ▼      ▼
read_0 read_1 read_2 read_3

首次输出（无 -y）：(read_0 - boot) / (t0 - boot)   ← 开机以来平均
第二次输出：       (read_1 - read_0) / 1             ← 第 1 秒内
第三次输出：       (read_2 - read_1) / 1             ← 第 2 秒内

加 -y 后：
第一次输出：       (read_1 - read_0) / 1             ← 第 1 秒内 ★
第二次输出：       (read_2 - read_1) / 1             ← 第 2 秒内
```

### Step 5: 诊断实战 — 从指标到结论

**目标：** 建立从 iostat 指标到诊断结论的思维链

**快速诊断流程图：**

```
iostat -x 1 观察
    │
    ├── %util ≈ 100%？
    │   ├── 是 → 设备饱和
    │   │   ├── await 高（> 20ms）→ I/O 延迟大，应用受影响
    │   │   │   ├── r_await 高 → 读延迟大，检查是否随机读
    │   │   │   └── w_await 高 → 写延迟大，检查队列深度
    │   │   └── await 低（< 5ms）→ 设备忙但响应快，不一定有问题
    │   └── 否 → 设备未饱和
    │
    ├── await 高但 %util 低？
    │   └── 可能是少量慢 I/O（大文件顺序读/写），不是瓶颈
    │
    ├── avgqu-sz 很大？
    │   └── 队列堆积 → I/O 调度器压力大
    │       └── 用 blktrace 看 Q2D 延迟
    │
    └── %iowait 高？
        ├── 是 → CPU 在等 I/O
        │   └── 检查 I/O 模式和文件系统
        └── 否 → CPU 不是被 I/O 阻塞
```

## 4. 核心概念

### %util 的本质

```
%util = 设备有 I/O 在忙的时间 / 总采样时间 × 100%

┌──────────────────────────────────────────┐
│ 1 秒内设备状态：                          │
│                                          │
│ ████░░░░████░░░░████░░░░████░░░░████░░░░ │
│ 忙  闲  忙  闲  忙  闲  忙  闲  忙  闲   │
│                                          │
│ 忙 = 600ms, 闲 = 400ms                   │
│ %util = 600/1000 × 100% = 60%            │
└──────────────────────────────────────────┘

数据来源：/proc/diskstats 的 io_ticks 字段
io_ticks = 设备有 I/O 在处理的 jiffies 数
```

**关键认知：** `%util` 不是"设备容量使用率"，而是"设备忙碌时间比"。

对于多队列 SSD：
- `%util=100%` 只说明"设备一直有 I/O 在处理"
- 不代表"设备已无法处理更多 I/O"
- 多队列设备可以同时处理多个 I/O，100% util 时可能还有大量剩余吞吐

### await 的组成

```
await = (io_wait_time) / (reads + writes)

io_wait_time = 所有 I/O 从提交到完成的总等待时间之和

await ≈ Q2C（端到端延迟）
     = 排队等待(Q2D) + 设备处理(D2C)
```

### 与 blktrace/btt 指标的对应关系

| iostat 指标 | blktrace/btt 对应 | 说明 |
|------------|------------------|------|
| `r/s` | IOPS（读） | 相同概念 |
| `w/s` | IOPS（写） | 相同概念 |
| `await` | Q2C avg | 端到端延迟 |
| `r_await` | 读请求的 Q2C | 仅读操作 |
| `w_await` | 写请求的 Q2C | 仅写操作 |
| `avgqu-sz` | btt 的 Average Queue Depth | 平均队列深度 |
| `%util` | 无直接对应 | blktrace 不计算利用率 |
| — | Q2D | iostat 不单独追踪排队延迟 |
| — | D2C | iostat 不单独追踪设备延迟 |

```bash
iostat -x 1 -y &
```

```bash
wait

# 分析 blktrace
blkparse -i trace -O -d trace.bin
btt -i trace.bin -o summary
```

## 6. ★ 性能诊断专题

### iostat 在性能诊断中的定位

iostat 是 **宏观 I/O 诊断的第一步**。它告诉你"设备忙不忙、延迟高不高"，但不告诉你"哪个 I/O 慢、为什么慢"。

```
┌─────────────────────────────────────────────────────────────────┐
│                    I/O 性能诊断分层                               │
│                                                                  │
│  第 1 层：宏观发现                                                │
│    iostat -x 1 → %util 高不高？await 大不大？                    │
│    ┌──────────────────────────────────────────────────┐          │
│    │ 发现异常 → 进入第 2 层                            │          │
│    │ 一切正常 → 瓶颈不在 I/O，转向 CPU/内存/网络分析   │          │
│    └──────────────────────────────────────────────────┘          │
│                                                                  │
│  第 2 层：进程定位                                                │
│    pidstat -d → 哪个进程的 kB_wr/s 或 kB_rd/s 最高？            │
│                                                                  │
│  第 3 层：微观追踪                                                │
│    blktrace + blkparse + btt → 每个 I/O 的 Q2D/D2C 延迟         │
│    Q2D 高 = 软件层瓶颈（调度器、队列深度）                        │
│    D2C 高 = 硬件层瓶颈（设备性能）                                │
└─────────────────────────────────────────────────────────────────┘
```

**★ 核心原则：** iostat 用于"发现问题"，blktrace 用于"定位问题"。

### ★ 诊断思维链 — 从 iostat 指标到结论

这是 iostat 诊断的**核心技能**。掌握以下思维链，可以快速判断 I/O 是否是瓶颈。

#### 场景 A：%util=100% 但 await 低

```
现象：
  %util = 100%     → 设备一直有 I/O 在处理
  await = 1~2ms    → 但每个 I/O 很快完成

诊断：
  设备饱和但响应快。常见于 SSD 满载场景。

  对于 SSD/NVMe：★ 可能是正常现象！
    SSD 的并行能力强，100% util 只说明"一直有活干"，
    不代表"干不过来了"。只要 await 低，应用不受影响。

  对于 HDD/virtio-blk（单队列）：
    100% util + 低 await = 设备接近吞吐上限但仍能及时响应。
    关注 avgqu-sz 是否在增长。

下一步：
  观察 avgqu-sz 是否持续增长。如果稳定，设备可以扛住。
  如果 avgqu-sz 在涨 → 即将过载，需要降低 I/O 压力。
```

#### 场景 B：%util=100% 且 await 高

```
现象：
  %util = 100%     → 设备一直忙
  await > 20ms     → 每个 I/O 等很久才完成

诊断：
  ★ 设备过载或存在故障。这是最典型的 I/O 瓶颈信号。

  可能原因：
  1. I/O 负载超过设备能力（IOPS 或吞吐超限）
  2. 设备降级（SSD 磨损、HDD 坏道）
  3. 写回风暴（脏页集中刷盘）
  4. I/O 调度器配置不当

下一步：
  1. 区分读/写延迟：r_await vs w_await
     - r_await 高 → 检查是否大量随机读
     - w_await 高 → 检查是否写回风暴或写放大
  2. 用 blktrace 分析 Q2D 和 D2C：
     - Q2D 高 → 软件层瓶颈（队列深度、调度器）
     - D2C 高 → 硬件层瓶颈（设备性能/故障）
  3. 用 pidstat -d 找到产生 I/O 最多的进程
```

#### 场景 C：%util 低但 await 高

```
现象：
  %util < 20%      → 设备大部分时间空闲
  await > 20ms     → 但偶尔的 I/O 耗时很长

诊断：
  偶发慢 I/O。设备没有持续负载，但某些 I/O 请求特别慢。

  常见原因：
  1. 大请求（rareq-sz/wareq-sz 很大，如 128KB+ 的顺序 I/O）
     → 单个大 I/O 的绝对延迟本来就高，但吞吐是正常的
  2. 偶发设备问题（SSD GC、HDD 磁头寻道）
  3. I/O 被调度器延迟（优先级问题）
  4. 设备健康下降（SMART 告警）

下一步：
  1. 看 rareq-sz / wareq-sz → 如果是大请求，await 高是正常的
  2. 看 r/s 和 w/s → 如果 IOPS 很低但 await 高，说明是少量慢 I/O
  3. 检查设备健康：smartctl -a /dev/sda
```

#### 场景 D：await 远大于 svctm

```
现象：
  await = 10ms     → 端到端延迟 10ms
  svctm = 0.5ms    → 设备实际处理只需 0.5ms

诊断：
  ★ 排队时间占了绝大部分！
  排队延迟 = await - svctm ≈ 9.5ms

  avgqu-sz 应该也很高（> 16）。
  说明 I/O 请求到达速率 > 设备处理速率，请求在队列中堆积。

下一步：
  1. 检查 avgqu-sz 确认排队严重
  2. 降低 iodepth 或 I/O 并发数
  3. 用 blktrace 看 Q2D 延迟分布

★ 注意：新版 sysstat（12+）已移除 svctm 列，
  因为 svctm 的计算假设不准确。
  推荐使用 await + avgqu-sz 组合判断。
```

### ★ 完整诊断决策树

```
iostat -x -y 1 观察
│
├─ 步骤 1：看 %util
│   ├─ %util ≈ 100%
│   │   ├─ await 低（< 5ms）→ ★ 场景 A：设备忙但响应快
│   │   │   └─ 观察 avgqu-sz 趋势
│   │   └─ await 高（> 20ms）→ ★ 场景 B：设备过载
│   │       └─ 区分 r_await / w_await → blktrace 深入
│   └─ %util 低（< 50%）
│       ├─ await 高 → ★ 场景 C：偶发慢 I/O
│       │   └─ 检查请求大小和设备健康
│       └─ await 低 → I/O 正常，瓶颈不在磁盘
│
├─ 步骤 2：看 await vs avgqu-sz
│   ├─ await 高 + avgqu-sz 高 → 排队严重（场景 D）
│   └─ await 高 + avgqu-sz 低 → 设备本身慢（D2C 高）
│
├─ 步骤 3：看 CPU 指标
│   ├─ %iowait 高 + %util 高 → I/O 确实是瓶颈
│   ├─ %iowait 高 + %util 低 → CPU 空闲，不是 I/O 瓶颈
│   └─ %iowait 低 → CPU 没有被 I/O 阻塞
│
└─ 步骤 4：看 r/s、w/s 和 rkB/s、wkB/s
    ├─ IOPS 高 + 小请求 → 随机 I/O 密集
    ├─ 吞吐高 + 大请求 → 顺序 I/O 密集
    └─ 都低 → I/O 负载不重
```

### CPU 高占用但 iostat 正常 — 瓶颈不在 I/O

```
典型场景：
  CPU 使用率 > 90%，但 iostat 显示：
  %util < 10%，await ≈ 0，r/s 和 w/s 都很低

诊断：
  ★ 瓶颈不在 I/O！CPU 高占用是由计算、锁竞争、或其他非 I/O 原因造成的。

下一步排查方向：
  ┌──────────────────────────────────────────────────────────┐
  │ 1. 找高 CPU 进程                                          │
  │    pidstat -u 1 → 按 %CPU 排序找消耗者                    │
  │                                                          │
  │ 2. 看上下文切换                                           │
  │    pidstat -w → cswch/s 高 = 锁等待                      │
  │                  nvcswch/s 高 = CPU 竞争                  │
  │                                                          │
  │ 3. 看软中断                                               │
  │    cat /proc/interrupts → 软中断是否集中在某 CPU           │
  │    mpstat -P ALL 1 → 各 CPU 的 %soft 分布                │
  │                                                          │
  │ 4. 看内存                                                 │
  │    free -m → 是否 swap 活跃                               │
  │    pidstat -r → majflt/s 高 = 缺页（内存压力）           │
  │                                                          │
  │ 5. perf / 火焰图 → 函数级热点分析                         │
  └──────────────────────────────────────────────────────────┘
```

### iostat 与 sar -d 对比

```
iostat 和 sar -d 的关系：

              iostat                        sar -d
              ──────                        ──────
数据来源      /proc/diskstats               /proc/diskstats（相同！）
输出字段      基本相同                       基本相同
历史记录      ❌ 只能实时观测                ✅ 可以记录历史数据
定时采集      ❌ 需要手动运行或后台脚本       ✅ sar 服务自动采集
事后分析      ❌ 不支持                      ✅ sar -d -f 文件
适用场景      实时诊断                       长期监控 + 事后回溯

★ 核心结论：
  两者数据完全相同（都来自 /proc/diskstats），区别在于：
  - iostat 是"现在进行时"工具（实时观测）
  - sar -d 是"过去完成时"工具（历史回溯）

实际工作流：
  1. 收到告警 → 用 sar -d -f 查看历史 I/O 趋势
  2. 确认问题时段 → 用 iostat -x 1 实时观测当前状态
  3. 实时定位 → blktrace 深入分析

sar 定时采集配置：
  # /etc/cron.d/sysstat — 每 10 分钟采集一次
  */10 * * * * root /usr/lib/sa/sa1 1 1

  # 查看今天的 I/O 历史
  sar -d

  # 查看指定日期的历史
  sar -d -f /var/log/sa/sa01
```

## 7. 与其他工具的协作

### iostat + blktrace 联合诊断流程

```
1. iostat -x 1 → 发现异常指标
   │
   ├── %util 持续 100%    → 设备饱和
   ├── await > 20ms       → I/O 延迟大
   ├── avgqu-sz > 32      → 队列堆积
   └── %iowait > 50%      → CPU 被 I/O 阻塞
   │
   ▼
2. blktrace -d /dev/vdb -o trace -w 30 → 追踪微观细节
   │
   ▼
3. blkparse → 看每个 I/O 的 Q/D/C 事件
   btt → 看 Q2D/D2C 延迟分布
   │
   ├── Q2D 高 → 软件层瓶颈（调度器、队列）
   └── D2C 高 → 硬件层瓶颈（设备性能）
```

### iostat + sar 长期监控

```bash
# sar 可以长期记录 iostat 数据
sar -d 1 3600 -o /tmp/sar_data     # 记录 1 小时的 I/O 数据

# 事后分析
sar -d -f /tmp/sar_data
```
