## 一、CPU 流水线总览

### 1.1 六阶段流水线

现代 CPU 将指令执行分解为多个阶段，形成流水线（Pipeline）：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    x86 CPU 六阶段流水线                                  │
├──────────┬──────────┬──────────┬──────────┬──────────┬────────────────┤
│  取指     │  解码     │  重命名   │  调度     │  执行     │  退休          │
│  Fetch   │  Decode  │  Rename  │  Schedule│  Execute │  Retire        │
│          │          │          │          │          │                │
│ 从 I-Cache│ x86指令  │ 消除寄存器│ 等待操作数│ ALU/FPU  │ 按序提交结果    │
│ 取出指令  │ → μop    │ 假依赖    │ 就绪则发射│ Load/Store│ 到架构状态     │
└──────────┴──────────┴──────────┴──────────┴──────────┴────────────────┘

  ★ 前端 ──────────────────★                    ★ 后端 ──────────────────★
  (Frontend)                                      (Backend)
```

**★ 关键概念：前端 vs 后端**

| 部分 | 阶段 | 职责 |
|------|------|------|
| **前端（Frontend）** | 取指 + 解码 + 重命名 | 将 x86 指令翻译为 μop，按序供给后端 |
| **后端（Backend）** | 调度 + 执行 + 退休 | 乱序执行 μop，按序退休结果 |

### 1.2 各阶段详解

#### 阶段 1：取指（Fetch）

```
┌─────────────────────────────────────────┐
│            Instruction Cache            │
│  ┌──────────────────────────────────┐   │
│  │  代码段（Text Segment）的缓存     │   │
│  │  L1-I: 32KB, 8-way, 64B line    │   │
│  └──────────────────────────────────┘   │
│           │                             │
│           ▼                             │
│  ┌──────────────────────────────────┐   │
│  │    Branch Prediction Unit       │   │
│  │    (分支预测单元)                 │   │
│  │  - 预测分支方向                   │   │
│  │  - 预测跳转目标                   │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘

取指带宽：每周期最多 16 字节（Skylake）
         ≈ 每周期 4-6 条 x86 指令
```

**★ 取指瓶颈来源：**
- I-cache miss：代码不在 L1-I 缓存中
- iTLB miss：虚拟地址→物理地址翻译失败
- 分支预测延迟：预测错误导致取指方向错误

#### 阶段 2：解码（Decode）

```
x86 指令（CISC）        μop（RISC-like）
┌──────────────┐      ┌──────────────┐
│ ADD rax, rbx │ ───→ │ add rax, rbx │  1 μop
├──────────────┤      ├──────────────┤
│ ADD [mem],rax│ ───→ │ load tmp,[m] │  2 μop（微融合）
│              │      │ add rax, tmp │
├──────────────┤      ├──────────────┤
│ REP MOVSB    │ ───→ │ μop_1        │  数十个 μop
│              │      │ μop_2        │  （微码展开）
│              │      │ ...          │
├──────────────┤      ├──────────────┤
│ CPUID        │ ───→ │ MS μop seq   │  微码辅助序列
└──────────────┘      └──────────────┘
```

**★ 解码带宽：**
- 4 个解码器：每周期最多解码 4 条简单指令（每条 ≤ 1 μop）
- MS（Microcode Sequencer）：复杂指令走微码 ROM，每周期产出较少 μop
- DSB（Decoded Stream Buffer / μop Cache）：已解码的 μop 缓存，跳过解码阶段

#### 阶段 3：重命名（Rename）

```
        架构寄存器（16 个 GPR）
        ┌───────┐
        │  RAX  │ ← 多个指令写 RAX？
        │  RBX  │    不需要等前一个完成！
        │  RCX  │
        │  ...  │
        └───┬───┘
            │ 重命名
            ▼
        物理寄存器堆（180+ 个）
        ┌───────────────────────┐
        │ P0  P1  P2  ... P179 │
        │ 消除假依赖（WAW/WAR） │
        └───────────────────────┘

★ 重命名的作用：消除寄存器假依赖，允许乱序执行
  - WAW（Write-After-Write）：两条指令写同一寄存器
  - WAR（Write-After-Read）：后一条指令写前一条指令读的寄存器
```

#### 阶段 4：调度（Schedule / Dispatch）

```
┌──────────────────────────────────────────────┐
│          Reservation Station (RS)            │
│          （保留站 / 发射队列）                  │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │  每个 μop 等待所有操作数就绪            │  │
│  │  就绪后立即发射到执行端口               │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  发射带宽：每周期最多 4 个 μop               │
│                                              │
│  执行端口（Skylake）：                        │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐  │
│  │ P0 │ P1 │ P2 │ P3 │ P4 │ P5 │ P6 │ P7 │  │
│  │ALU │ALU │Load│Store│Store│ALU │ALU │Store│ │
│  │Br  │Mul │    │Addr │Data│Shft│ALU │Addr│ │
│  └────┴────┴────┴────┴────┴────┴────┴────┘  │
│                                              │
│  ★ 8 个执行端口，但每周期最多发射 4 个 μop    │
└──────────────────────────────────────────────┘
```

#### 阶段 5：执行（Execute）

```
μop 类型          延迟（周期）    执行端口
─────────────────────────────────────────────
ADD/SUB           1              P0,P1,P5,P6
MUL               3              P1
DIV (64-bit)      35-90          P0
Load (L1 hit)     4              P2,P3
Load (L2 hit)     12             P2,P3
Load (L3 hit)     42             P2,P3
Load (DRAM)       180+           P2,P3
Branch            1              P0,P6

★ 关键：Load 的延迟差异巨大（4 vs 180+ 周期）
  这就是 Memory Bound 的根源
```

#### 阶段 6：退休（Retire）

```
┌──────────────────────────────────────────────┐
│         Reorder Buffer (ROB)                 │
│         （重排序缓冲区）                       │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │  指令按发射顺序进入 ROB                  │  │
│  │  执行可能乱序完成                         │  │
│  │  退休必须按原始顺序提交                   │  │
│  │                                          │  │
│  │  ROB 大小：                              │  │
│  │  Skylake: 224 条目                       │  │
│  │  Ice Lake: 352 条目                      │  │
│  │  Sapphire Rapids: 512 条目               │  │
│  └────────────────────────────────────────┘  │
│                                              │
│  ★ 退休带宽：每周期最多 4 个 μop             │
│  ★ 这就是 Pipeline Width = 4 的来源          │
└──────────────────────────────────────────────┘
```

## 二、Pipeline Slot 详解

### 2.1 定义

```
Pipeline Slot = 一个 μop 在一个时钟周期内占据的一个退休位置

Total Slots = Pipeline_Width × CPU_CLK_UNHALTED.THREAD
            = 4 × 总周期数
```

### 2.2 图解：Slot 的填充

```
         Cycle 1    Cycle 2    Cycle 3    Cycle 4    Cycle 5
         ─────────  ─────────  ─────────  ─────────  ─────────
Slot 0   [ADD  ]   [CMP  ]   [------]   [MOV  ]   [SUB  ]
Slot 1   [MUL  ]   [JMP  ]   [------]   [ADD  ]   [XOR  ]
Slot 2   [------]   [------]   [------]   [LOAD ]   [------]
Slot 3   [------]   [------]   [------]   [------]   [------]
         ─────────  ─────────  ─────────  ─────────  ─────────
填充率    2/4       2/4        0/4        3/4        2/4
         (50%)     (50%)      (0%)       (75%)      (50%)

★ Cycle 3 完全空闲 → Pipeline Stall
★ 5 个周期 20 个 Slot 中只填充了 9 个 → 45% 利用率
```

### 2.3 为什么有空白 Slot？

空白 Slot 被归入四个互斥的类别：

```
┌────────────────────────────────────────────────────────────┐
│                  Total Pipeline Slots                      │
│                                                            │
│   ┌────────────────────────────────────────────────────┐   │
│   │ Retiring: 成功退休的 μop 占据的 Slot               │   │
│   │ → 这是有效工作                                      │   │
│   └────────────────────────────────────────────────────┘   │
│                                                            │
│   ┌────────────────────────────────────────────────────┐   │
│   │ Frontend Bound: 前端未能提供足够 μop 导致的空 Slot  │   │
│   │ → "喂不饱"                                         │   │
│   └────────────────────────────────────────────────────┘   │
│                                                            │
│   ┌────────────────────────────────────────────────────┐   │
│   │ Backend Bound: 后端无法处理 μop 导致的空 Slot       │   │
│   │ → "消化不了"                                        │   │
│   └────────────────────────────────────────────────────┘   │
│                                                            │
│   ┌────────────────────────────────────────────────────┐   │
│   │ Bad Speculation: 错误推测后丢弃的 μop + 空 Slot    │   │
│   │ → "走弯路"                                          │   │
│   └────────────────────────────────────────────────────┘   │
│                                                            │
│   ★ 四类之和 = Total Slots（100%）                         │
└────────────────────────────────────────────────────────────┘
```

## 三、四大类瓶颈详解

### 3.1 Retiring（有效退休）

**★ 唯一"好"的类别——代表有效工作**

```
Retiring = INST_RETIRED.ANY / Total_Slots

含义：成功执行并提交了结果的 μop 占总 Slot 的比例
```

**Retiring 的二级分类：**

```
Retiring
├── Base（基本退休）
│   → 正常的单 μop 指令退休
│   → 如 ADD, SUB, MOV, CMP
│
└── Microcode Sequencer（微码辅助退休）
    → 复杂指令通过微码序列退休
    → 如 REP MOVSB, CPUID, x87 浮点
    → ★ 通常意味着可以优化的复杂指令
```

**Retiring 占比解读：**

| 占比 | 评价 | 说明 |
|------|------|------|
| > 70% | ★ 优秀 | IPC 接近上限 |
| 50-70% | 良好 | 正常范围 |
| 30-50% | 一般 | 有优化空间 |
| < 30% | ★ 差 | 严重瓶颈，需下钻分析 |

**优化方向：** 提高 Retiring 不是直接优化的目标——它是其他三类降低后的自然结果。

### 3.2 Frontend Bound（前端瓶颈）

**★ "喂不饱"——前端无法按时提供足够的 μop 给后端**

```
前端流水线：
  取指 ─→ 解码 ─→ 重命名 ─→ 送入后端
    │        │        │
    │        │        └── 重命名后 μop 进入发射队列
    │        └── x86 → μop 翻译
    └── 从 I-Cache 取出指令字节

如果前端某一步卡住，后端就"饿"了
→ 后端可用的 μop 不够填满 Slot
→ 这些空 Slot 就是 Frontend Bound
```

**Frontend Bound 的二级分类：**

```
Frontend Bound
├── Fetch Latency（取指延迟）
│   → 前端某个阶段延迟导致 μop 供给中断
│   → ★ 主要原因：
│     · I-Cache Miss：代码不在 L1-I 中
│     · iTLB Miss：指令地址翻译失败
│     · Branch Resteers：分支预测纠正后的重新取指
│     · DSB Switches：μop Cache 与常规解码器切换
│
└── Fetch Bandwidth（取指带宽）
    → 前端能工作但带宽不够
    → ★ 主要原因：
      · 解码器瓶颈：每周期只能解码 4 条简单指令
      · MS 瓶颈：微码序列每周期产出更少 μop
      · 大指令（长编码）：一条指令占多个字节
```

**Frontend Bound 的典型场景：**

| 场景 | 原因 | 优化方法 |
|------|------|---------|
| 代码量巨大（> 32KB 热代码） | I-Cache 容量不足 | 减少热路径代码量，LTO 优化 |
| 大量分支 | 分支预测器频繁更新 | 减少分支，使用 likely/unlikely |
| 大量短函数 | 取指跨页导致 iTLB miss | Huge Pages for text |
| 复杂指令（REP MOVSB 等） | 走微码路径 | 使用简单指令替代 |

### 3.3 Bad Speculation（错误推测）

**★ "走弯路"——CPU 做了无用功**

```
正常执行：
  if (x > 0) {     ← 分支预测：会进入
    do_something();  ← 推测执行
  }
  → 预测正确 ✓ → μop 正常退休

错误推测：
  if (x > 0) {     ← 分支预测：会进入
    do_something();  ← 推测执行了！
  }
  → 实际 x <= 0 ✗ → 推测的 μop 全部丢弃
  → 流水线冲刷（Pipeline Flush）
  → 浪费的 Slot = 推测执行的 μop + 冲刷期间的空 Slot
```

**Bad Speculation 的二级分类：**

```
Bad Speculation
├── Branch Mispredict（分支预测错误）
│   → 预测了错误的分支方向或目标
│   → ★ 占 Bad Speculation 的绝大部分（通常 > 90%）
│   → 惩罚：15-20 个周期的流水线冲刷（Skylake）
│
└── Machine Clears（流水线完全冲刷）
    → 比分支预测错误更严重
    → 整个流水线必须清空重来
    → ★ 主要原因：
      · Memory Ordering Violation（内存排序违规）
      · Self-Modifying Code（自修改代码）
      · SMC/SNOOP 命中
```

**Bad Speculation 的典型场景：**

| 场景 | 原因 | 优化方法 |
|------|------|---------|
| 数据依赖的分支（如 `if(data[i]>threshold)`） | 分支模式不可预测 | 排序数据、CMOV、查表 |
| 间接跳转（虚函数调用） | 目标地址难以预测 | 减少多态、使用 switch |
| 稀疏的 switch-case | 预测器无法学习模式 | 重新排序 case |

**★ 经典案例：排序 vs 未排序数组**

```c
// 未排序：分支预测错误率 ≈ 50%
int data[] = {随机值...};
for (int i = 0; i < N; i++)
    if (data[i] > threshold) sum += data[i];
// Bad Speculation ≈ 30-40%

// 排序后：分支预测几乎完美
qsort(data, N, ...);
for (int i = 0; i < N; i++)
    if (data[i] > threshold) sum += data[i];
// Bad Speculation ≈ 1-2%
```

### 3.4 Backend Bound（后端瓶颈）

**★ "消化不了"——后端执行能力不足**

```
后端流水线：
  发射队列 → 执行端口 → 结果写回 → ROB 退休
     │          │          │
     │          │          └── 写回 ROB
     │          └── ALU、Load/Store、乘法、除法
     └── 等待操作数就绪

如果后端某个步骤卡住：
→ μop 堆积在 ROB 中
→ 前端无法继续送入新 μop（ROB 满）
→ 这些空 Slot 就是 Backend Bound
```

**Backend Bound 的二级分类：**

```
Backend Bound
├── Memory Bound（内存瓶颈）
│   → Load/Store 操作等待内存子系统
│   → ★ 主要原因：
│     · L1 D-Cache Miss（4 周期 → 12 周期）
│     · L2 Cache Miss（12 周期 → 42 周期）
│     · L3 Cache Miss（42 周期 → 180+ 周期）
│     · DRAM 带宽饱和
│     · Store Buffer 满
│
└── Core Bound（执行单元瓶颈）
    → 执行端口不够用或指令延迟长
    → ★ 主要原因：
      · 长延迟指令（除法：35-90 周期）
      · 执行端口竞争（所有 μop 都要用 P0）
      · 指令级并行度（ILP）不足
      · 依赖链过长
```

**Memory Bound vs Core Bound 的区分：**

```
Memory Bound 特征：
┌────────────────────────────────────┐
│  μop: LOAD [addr]                  │
│  等待 L1 miss → L2 miss → L3 miss  │
│  ───────────────────── 等 180 周期  │
│                                    │
│  ROB 中的后续 μop 全部等待          │
│  → 后端停滞，前端也停滞              │
└────────────────────────────────────┘

Core Bound 特征：
┌────────────────────────────────────┐
│  μop: DIV rax, rbx                 │
│  执行中，占 P0 端口 90 个周期        │
│  ───────────────────── 占 90 周期   │
│                                    │
│  其他 μop 可以用 P1-P7             │
│  但如果后续也依赖 P0 → 端口竞争     │
└────────────────────────────────────┘
```

**Backend Bound 的典型场景：**

| 场景 | 子类 | 原因 | 优化方法 |
|------|------|------|---------|
| 链表遍历 | Memory | 指针追逐，无法预取 | 数组化、预排序 |
| 矩阵乘法（朴素） | Memory | 缓存不友好 | 分块、转置 |
| 大量除法 | Core | 除法延迟极高 | 乘以倒数 |
| 顺序内存拷贝 | Memory | 带宽饱和 | 向量化、多线程 |
| 递归斐波那契 | Core | 长依赖链 | 动态规划、循环 |

## 四、完整的流水线 Slot 分配图

### 4.1 一个周期内的 Slot 分配决策

```
                    本周期是否有 μop 可退休？
                           │
                    ┌──────┴──────┐
                    │ YES         │ NO
                    ▼             ▼
            ┌──────────┐    为什么没有？
            │ Retiring │    ┌──────┴──────┐
            │ 计数 +1  │    │             │
            └──────────┘    ▼             ▼
                     前端饥饿？      后端卡住？
                        │               │
                   ┌────┴────┐     ┌────┴────┐
                   │ YES     │     │ YES     │
                   ▼         │     ▼         │
             Frontend Bound  │  Backend Bound │
             计数 +剩余Slot  │  计数 +剩余Slot│

                    │ 供给了μop  │            │

                          │                  │
                          ▼                  │
                  Bad Speculation            │

### 4.2 四类瓶颈的时间线对比

```

R R R R B B  R R R R B B  R R R R B B  R R R R
（R=Retiring, B=Backend Bound, ░=空闲）

Frontend Bound 高的程序（如大代码量）：
░░░░░░░░░░░░████░░░░░░░░░░░░████░░░░░░░░░░░░████
F F F F F F  R R  F F F F F F  R R  F F F F F F
（F=Frontend Bound, R=Retiring）

Backend Bound 高的程序（如内存密集）：

## 五、Pipeline Slot 与 IPC 的关系

### 5.1 数学关系

```
IPC = INST_RETIRED.ANY / CPU_CLK_UNHALTED.THREAD

Total_Slots = 4 × CPU_CLK_UNHALTED.THREAD

Retiring% = INST_RETIRED.ANY / Total_Slots
          = INST_RETIRED.ANY / (4 × CPU_CLK_UNHALTED.THREAD)
          = IPC / 4

★ 因此：IPC = Retiring% × 4（Pipeline Width）
```

### 5.2 IPC 与 Retiring% 对照表

| IPC | Retiring% | 浪费率 | 评价 |
|-----|-----------|--------|------|
| 4.0 | 100% | 0% | 理论极限（不可能达到） |
| 3.0 | 75% | 25% | 非常优秀 |
| 2.0 | 50% | 50% | 良好 |
| 1.0 | 25% | 75% | 一般 |
| 0.5 | 12.5% | 87.5% | ★ 差，需要优化 |

## 六、从 x86 指令到 μop 的转换

### 6.1 常见指令的 μop 分解

```
x86 指令                  → μop 分解                      μop数
─────────────────────────────────────────────────────────────────
ADD r64, r64             → add r64, r64                   1
ADD r64, [mem]           → load tmp, [mem]                2 (微融合)
                         → add r64, tmp
ADD [mem], r64           → load tmp, [mem]                3
                         → add tmp, r64
                         → store [mem], tmp
PUSH r64                 → sub rsp, 8                     2 (微融合)
                         → store [rsp], r64
CALL func                → push rip                        3+
                         → jmp func
RET                      → pop tmp                         2+
                         → jmp tmp
REP MOVSB                → MS 微码序列                     100+
                         → (循环: load + store + 指针更新)
CPUID                    → MS 微码序列 (序列化)             50+
```

### 6.2 微融合（Micro-fusion）与宏融合（Macro-fusion）

```
微融合（Micro-fusion）：
  ADD rax, [rbx]
  → 在 ROB 中占 1 个条目（融合为 1 个 μop）
  → 在发射队列中占 1 个条目
  → ★ 直到执行时才拆分为 load + add

宏融合（Macro-fusion）：
  CMP rax, rbx         ← 比较
  JE  label            ← 条件跳转
  → 融合为 1 个 μop
  → ★ 减少退休带宽压力

★ 融合是提高 Retiring% 的重要机制
```

## 八、总结：TMAM 分析思维

### 8.1 分析流程

```
┌──────────────────────────────────────────────────────────────┐
│                    TMAM 分析流程                              │
│                                                              │
│  1. 采集 Level 1 数据                                         │
│     perf stat --topdown --td-level 1 -- ./program            │
│     │                                                        │
│  2. 找到占比最大的类别                                         │
│     │                                                        │
│     ├── Retiring 最高 → 程序效率高，无需优化                    │
│     ├── Frontend Bound → 下钻 Level 2: Fetch Lat/BW          │
│     ├── Bad Speculation → 下钻 Level 2: Branch Mispre/MC     │
│     └── Backend Bound → 下钻 Level 2: Memory/Core Bound      │
│                                                              │
│  3. 继续下钻到 Level 3/4                                      │
│     找到具体的子原因                                           │
│                                                              │
│  4. 针对性优化                                                │
│     └── 验证优化效果（重新采集 topdown）                       │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 关键原则

| 原则 | 说明 |
|------|------|
| ★ 从顶层开始 | 永远先看 Level 1，不要跳级 |
| ★ 关注最大项 | 优化占比最大的瓶颈收益最高 |
| ★ 百分比优先 | 绝对数值受运行时间影响，百分比更稳定 |
| ★ 对比分析 | 优化前后对比 topdown 分布 |
| ★ 结合其他工具 | topdown 告诉你"在哪"，perf record 告诉你"谁" |

## 九、思考题

1. **如果 IPC = 2.0，是否意味着 50% 的 Slot 被浪费了？**
   > 是的。IPC = 2.0 时 Retiring% = 2.0/4 = 50%。但 50% 的浪费不一定都在同一类——可能 Frontend 10%、Backend 30%、Bad Spec 10%。

2. **为什么不能直接优化 Retiring？**
   > Retiring 是"好的"部分。你应该优化其他三类——当 Frontend Bound 从 20% 降到 5% 时，释放出的 Slot 自然被 Retiring 填充。

3. **超线程（SMT）如何影响 TMAM？**
   > 超线程让两个逻辑线程共享一个物理核心的执行资源。每个线程有独立的 TMAM 分析，但它们共享 Pipeline Slot 池——一个线程的 stall 会被另一个线程的 μop 填充。

4. **为什么 Backend Bound 中 Memory Bound 通常远大于 Core Bound？**
   > 因为内存访问的延迟层次极大（L1=4 周期 vs DRAM=180+ 周期），而 CPU 的 ALU 操作通常只需 1 个周期。现代 CPU 的计算能力远超内存子系统的供给能力——这就是所谓的"内存墙"（Memory Wall）。
