# 可观测性

## 1. 可观测性三支柱

```
┌─────────────────────────────────────────┐
│           可观测性三支柱                   │
├──────────┬──────────┬────────────────────┤
│  日志    │  指标    │  追踪              │
│  Logs    │  Metrics │  Traces           │
│          │          │                    │
│ 离散事件 │ 聚合数值 │ 请求全链路          │
│ 事后排查 │ 实时监控 │  分布式调用关系     │
└──────────┴──────────┴────────────────────┘

★ 可观测性 vs 监控：
  监控 = "告诉你有问题"（被动告警）
  可观测性 = "让你能查清为什么"（主动探索）
  → 监控是可观测性的子集
```

## 2. 结构化日志

### 为什么结构化比 printf 好

```
传统日志（不可解析）：
  printf("user %s logged in from %s\n", user, ip);
  → 只能人眼读，无法程序化分析

结构化日志（JSON）：
  {"ts":"2026-07-04T10:00:00Z","level":"INFO","msg":"login",
   "user":"alice","ip":"1.2.3.4","request_id":"abc123"}
  → 可用 jq/ELK/Grafana Loki 程序化分析
```

### 日志级别的使用原则

```
FATAL：系统无法继续运行，必须人工干预
  例：无法打开数据文件、WAL 损坏

ERROR：业务出错，需要关注
  例：请求处理失败、连接被拒

WARN：异常但可恢复，潜在问题
  例：重试成功、接近资源上限

INFO：关键业务节点（低频！）
  例：服务启动、配置变更、请求完成统计

DEBUG：调试细节（生产默认关）
  例：进入函数、变量值

TRACE：最细粒度（几乎不用）
  例：每条 SQL、每个 I/O

★ 生产环境：INFO+WARN+ERROR+FATAL
  排查问题：临时开 DEBUG
```

### 关键字段设计

```
每条日志必须包含：
  timestamp    — 时间戳（ISO8601 或 epoch）
  level        — 日志级别
  request_id   — ★ 请求唯一 ID（贯穿一个请求的所有日志）
  trace_id     — ★ 分布式追踪 ID（贯穿全链路）
  message      — 人类可读描述

可选字段：
  user_id      — 用户标识
  latency_us   — 操作延迟（微秒）
  error_code   — 错误码
  component    — 组件名
```

### 日志采样

```
高频事件不要每条都记（日志风暴）：

  每秒 100 万次 I/O → 如果每次都记日志 → 磁盘 I/O 被日志占满

采样策略：
  - 每秒最多记 100 条
  - 或按比例采样（1% 请求记详细日志）
  - ★ 错误日志不采样（每条都记）
```

## 3. 指标（Metrics）

### 四种指标类型

```
Counter（计数器）：单调递增
  例：总请求数、总错误数
  → 只增不减，适合 rate() 计算

Gauge（瞬时值）：可增可减
  例：当前连接数、队列深度、内存使用
  → 反映当前状态

Histogram（直方图）：分布统计
  例：延迟分布（P50/P90/P99/P999）
  → ★ 比平均值重要得多

Summary（摘要）：客户端计算分位数
  例：Go 的 Prometheus Summary
  → 与 Histogram 类似但计算位置不同
```

### USE 方法（资源指标）

```
每个资源检查三个维度：

  U（Utilization）：使用率
    例：CPU 80%、磁盘空间 90%

  S（Saturation）：饱和度（排队）
    例：run queue 长度、avgqu-sz、等待的请求数

  E（Errors）：错误
    例：EIO 错误、重传、丢包

  ★ 三者组合判断：
    U 高 + S 低 = 正常满载
    U 高 + S 高 = 过载（需要扩容）
    U 低 + E 高 = 异常（检查配置）
```

### RED 方法（服务指标）

```
每个服务检查三个维度：

  R（Rate）：请求速率（QPS）
  E（Errors）：错误率
  D（Duration）：延迟分布（P99）

  ★ RED 适合微服务/API 的监控
  ★ USE 适合基础设施（CPU/磁盘/网络）
```

### 为什么 P99 比平均值重要

```
请求延迟分布：
  99 个请求：1ms
  1 个请求：100ms

  平均值 = (99×1 + 1×100)/100 = 1.99ms → 看起来很好！
  P99 = 100ms → ★ 揭示了长尾问题

  ★ 用户关心的是"最差体验"，不是"平均体验"
  → 监控 P99/P999 比平均值有意义
```

## 4. 分布式追踪

```
OpenTelemetry / Jaeger / Zipkin

请求全链路追踪：

  Client → Service A → Service B → DB
    │         │           │         │
    span1     span2       span3    span4
    │         │           │         │
    └─────────┴───────────┴─────────┘
         trace_id 贯穿全链路

  trace_id = "abc123"（整个请求唯一）
  span_id = 每个操作唯一
  parent_span_id = 调用者的 span_id（构建调用树）

  ★ 价值：能看到"请求在哪个服务耗时最多"
  ★ 采样：不是每个请求都追踪（开销大）
    - 头部采样：入口决定是否采样
    - 尾部采样：出口决定（能看到错误才采样）→ 更聪明但复杂
```

## 5. 调试辅助

### 信号触发调试

```c
// ★ 生产环境用 SIGUSR1/SIGUSR2 触发状态 dump
void sigusr1_handler(int sig) {
    // dump 内存状态到文件
    FILE *f = fopen("/tmp/dump.txt", "w");
    dump_state(f);  // 自定义：输出缓存内容、队列状态、连接数等
    fclose(f);
}

// ★ FUSE 的 -d（debug）模式：输出每个 opcode
// 生产不方便用（日志太多），但开发调试时很有用
```

### core dump 分析

```bash
# 开启 core dump
ulimit -c unlimited
echo "/tmp/core.%p" > /proc/sys/kernel/core_pattern

# 崩溃后分析
gdb /path/to/binary /tmp/core.12345
(gdb) bt          # 调用栈
(gdb) info threads # 线程
(gdb) print var    # 查看变量
```

## 6. 健康检查

```
liveness（存活检查）：
  "进程还活着吗"
  → 不活 → 重启
  → 简单：HTTP 200 或 TCP 端口

readiness（就绪检查）：
  "能处理请求吗"
  → 不就绪 → 从负载均衡摘除（但不重启）
  → 深度检查：依赖是否正常、是否还在恢复

  ★ 区别：
    liveness 失败 → 重启
    readiness 失败 → 摘除流量（可能只是临时忙）
```

## 7. 存储系统的可观测性

```
存储系统特有指标：

  IOPS：每秒 I/O 数（分读/写）
  带宽：MB/s（分读/写）
  延迟：P50/P99/P999（分读/写）
  队列深度：avgqu-sz
  ★ fsync 延迟：★ 存储系统最关键指标
    → fsync 延迟 = 数据落盘延迟 = 崩溃风险窗口
  空间使用：已用/可用/预测用满时间
  错误率：EIO/ENOSPC/timeout
```

## 8. 告警设计

```
告警疲劳问题：
  → 告警太多 → 运维麻木 → 真正的告警被忽略
  → ★ 原则：每个告警都应该需要人工行动

SLO-based 告警：
  定义 SLO："99% 的请求 < 100ms"
  错误预算 = 1% × 时间
  → 消耗错误预算过快 → 告警
  → ★ 基于长期趋势，不是单次抖动

告警分级：
  P0：立即响应（5 分钟）— 服务全挂、数据丢失风险
  P1：尽快响应（1 小时）— 部分功能降级
  P2：工作时间响应 — 潜在问题、容量预警
```
