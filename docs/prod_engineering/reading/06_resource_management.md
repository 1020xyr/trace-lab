# 资源管理与反压

## 1. 常见资源与上限

```
资源          上限                耗尽后果
─────────    ────────────────    ─────────────
内存          物理内存             OOM kill
文件描述符     RLIMIT_NOFILE       无法 open/socket → "too many open files"
线程          线程数上限           无法创建线程
CPU           核数                 CPU 100% → 响应慢
磁盘空间      容量                 ENOSPC → 无法写入
inode         文件系统 inode 数    ENOSPC（有空间但无 inode）
端口          65535（+ TIME_WAIT） 无法 bind → "address already in use"
I/O 带宽      设备吞吐             I/O 排队 → 延迟升高
```

## 2. 内存池

```
问题：高频 malloc/free → 碎片 + 分配器锁竞争

解决：预分配 + 池化

  启动时：malloc N 个对象 → 放入池
  运行时：从池取 → 用完归还
  ★ 无运行时 malloc → 无锁竞争、无碎片

  fio 的 io_u 池就是这个机制（iodepth 个预分配）

内存泄漏检测：
  valgrind --leak-check=full ./program
  或 AddressSanitizer：gcc -fsanitize=address
```

## 3. fd 池与连接池

```
问题：每个请求都 open/close → 系统调用开销 + fd 耗尽

解决：连接池/fd 池

  启动时：预建 N 个连接 → 池
  请求时：从池借 → 用完归还
  ★ 复用连接 → 减少系统调用 + 控制 fd 总数

  ★ 注意：连接池大小要合理
    太小 → 请求排队等连接
    太大 → 连接占资源 + 下游过载
```

## 4. 限流（Rate Limiting）

```
保护下游不被打爆：

  令牌桶（token bucket）：
    每秒产生 N 个令牌 → 请求消耗令牌
    → 平均速率 = N，允许突发（桶里积累的令牌）

  漏桶（leaky bucket）：
    请求入桶 → 匀速漏出
    → 严格匀速，无突发

  ★ 存储 I/O 限流：
    限制每个客户端的 IOPS/带宽
    → 防止单客户端占满设备
```

## 5. OOM 处理

```
OOM 是真实的生产威胁：

  内存泄漏 → 逐渐增长 → 触发 OOM killer
  → 内核杀进程（可能杀关键进程）

  设计策略：
    1. 监控内存使用 → 接近上限告警
    2. 主动降级（丢弃缓存/拒绝新请求）
    3. ★ 优雅处理 malloc 失败（不要忽略返回值）
    4. 设置内存上限（cgroups memory.limit）
    5. OOM 时优先杀非关键进程（oom_score_adj）

  ★ 存储系统：限制缓存大小
    → 不要无限缓存 → 设上限 + LRU 淘汰
```

## 6. cgroups 资源限制

```
cgroups 隔离资源：

  cpu.shares     → CPU 权重
  memory.limit   → 内存上限
  blkio.throttle → I/O 带宽/ IOPS 限制
  pids.max       → 进程数上限

  ★ 多租户隔离：
    每个租户一个 cgroup → 资源不互相干扰
    → 防止"吵闹的邻居"
```
