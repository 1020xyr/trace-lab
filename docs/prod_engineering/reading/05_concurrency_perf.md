# 并发与性能工程

## 1. 锁策略选择

```
按临界区大小选择：
  临界区极短（<几十纳秒）→ spinlock（忙等）
  临界区中等            → mutex（可睡眠）
  读多写少              → rwlock（读写分离）
  读写都极频繁          → RCU（读无锁）

  ★ 锁的粒度：
    粗锁（全局锁）→ 简单但并发低
    细锁（per-object/per-cpu）→ 并发高但复杂
    → 平衡：从粗到细逐步优化
```

## 2. 无锁编程

```
适用场景：计数器、简单状态

  CAS（Compare-And-Swap）：
    while (1) {
        old = *ptr;
        new = old + 1;
        if (CAS(ptr, old, new))  // 原子比较交换
            break;
    }
  
  ★ 无锁 vs 有锁：
    无锁：低竞争时快，高竞争时 CAS 失败多 → 反而慢
    有锁：竞争时线程睡眠 → 不占 CPU
    → 低竞争用无锁，高竞争用锁（或队列）

  per-CPU 数据结构（Linux 常用）：
    每个 CPU 独立的计数器 → 无锁
    汇总时遍历所有 CPU → 短暂加锁
```

## 3. 批量化（Batching）

```
单条处理 vs 批量处理：

  单条：
    for (i = 0; i < 1000; i++)
        write(fd, &buf[i], 1);  // ★ 1000 次系统调用
    
  批量：
    write(fd, buf, 1000);  // ★ 1 次系统调用
    
  性能差异：1000x

  存储系统的批量化：
    - io_submit 批量提交（libaio）
    - WAL 批量 fsync（group commit）
    - 网络批量发送（GSO/TSO）
    - 批量分配（slab 的 per-CPU 缓存）
```

## 4. 异步 I/O 模式

```
同步 I/O：
  submit → 阻塞等待 → 完成 → 下一个
  → QD=1，吞吐低

异步 I/O（Reactor 模式）：
  submit → 立即返回 → 做其他事 → 完成通知 → 处理
  → QD>1，吞吐高

  事件循环（epoll/io_uring）：
    while (1) {
        events = epoll_wait(epfd, ...);  // 等待事件
        for (e in events)
            handle(e);  // 处理完成
    }
  ★ Nginx/nginx 的核心模型

  ★ io_uring：Linux 现代异步 I/O
    SQ（提交队列）+ CQ（完成队列）
    无需系统调用提交（SQE 共享内存）
```

## 5. 避免常见并发陷阱

```
活锁（livelock）：
  → 线程不断重试但无法进展
  例：多个线程同时 CAS 失败 → 不断重试 → CPU 100% 但无进展
  → 解决：加随机退避

饥饿（starvation）：
  → 低优先级线程永远拿不到资源
  → 解决：公平锁（FIFO）或动态优先级

优先级反转：
  → 低优先级持锁 → 高优先级等待 → 中优先级抢占低 → 死锁感
  → 解决：优先级继承

★ false sharing：
  多核写同一 cache line → 性能暴降
  → 解决：cache line 对齐（aligned(64)）
```

## 6. 性能反模式

```
要避免的反模式：

  1. 锁粒度过粗
     全局锁保护所有操作 → 多核退化成单核
     
  2. 锁临界区有 I/O
     持锁期间做 fsync → 锁持有时间 = 磁盘延迟
     
  3. 同步 I/O 在热路径
     每个请求都 fsync → 吞吐被 fsync 延迟限制
     
  4. 无限制队列
     队列无限大 → 内存耗尽 → OOM
     
  5. 忽略 NUMA
     跨 NUMA 访问内存 → 延迟翻倍
     → 绑定 CPU + 本地内存分配
```
