# 一致性与分布式

## 1. CAP 定理

```
分布式系统三选二：
  C（Consistency）：一致性（所有节点同一时刻看到相同数据）
  A（Availability）：可用性（每个请求都有响应）
  P（Partition tolerance）：分区容错（网络分区时仍工作）

  ★ 网络分区（P）是必然的 → 实际是 C 和 A 二选一：
    CP：分区时拒绝写（保证一致）→ ZooKeeper/etcd
    AP：分区时接受写（保证可用）→ Cassandra/DynamoDB
    → 存储系统通常选 CP（数据不能不一致）

  ★ 最终一致性（AP 的折中）：
    分区时各写各的 → 分区恢复后合并
    → 合并时可能冲突 → 需要冲突解决策略
```

## 2. 共识协议

```
Raft/Paxos：多数派共识

  写入流程（Raft）：
    1. Leader 收到写请求
    2. Leader 发 AppendEntries 给所有 Follower
    3. ★ 多数（N/2+1）确认 → Leader commit
    4. Leader 返回成功给客户端
    5. Leader 通知 Follower commit

  ★ 为什么多数派：
    3 节点 → 容忍 1 节点故障（2/3 多数）
    5 节点 → 容忍 2 节点故障（3/5 多数）
    → 写入需要多数 → 总能选出唯一 Leader

  etcd/ZooKeeper/Consul 用 Raft/Paxos
```

## 3. 幂等与 fencing

```
分布式系统的重复请求问题：
  网络超时 → 客户端重试 → 服务端可能执行两次

  幂等：同一操作多次执行，结果一致
    → 用唯一 request_id 去重

  fencing（栅栏）：
    解决"僵尸 Leader"问题
    Leader A 网络分区 → 选出新 Leader B
    A 恢复后还在发旧请求 → 用 fencing token 拒绝

    fencing token 单调递增：
      A 的 token = 5
      B 选上 → token = 6
      A 发请求带 token=5 → 服务端拒绝（< 6）
      ★ 防止旧 Leader 损坏数据
```

## 4. 脑裂（Split-Brain）

```
网络分区导致两个分区各自选 Leader：

  分区 1: A（认为自己是 Leader）
  分区 2: B（认为自己是 Leader）

  → 两个 Leader 同时写 → 数据冲突

  解决：多数派 quorum
    3 节点分区为 1+2
    → 只有 2 节点的分区能凑够多数（2/3）
    → 1 节点的分区凑不够 → 不接受写
    ★ 只有能凑够多数的分区才服务
```

## 5. 分布式存储的一致性

```
副本一致性：
  写入 → 需要多少副本确认？
  
  W（写入副本数）/ R（读取副本数）/ N（总副本数）
  
  W + R > N → 强一致（读写必有交集）
  W + R <= N → 弱一致（可能读到旧数据）

  例：3 副本，W=2, R=2
    写：2 副本确认
    读：读 2 副本取最新
    → W+R=4 > N=3 → 强一致

  ★ 存储系统常用：
    Quorum 写（W=2/3）→ 性能和安全的平衡
    全部确认（W=3/3）→ 最安全但慢
```

## 6. 分布式事务

```
跨节点事务的挑战：
  节点 A 写 + 节点 B 写 → 要么都成功，要么都失败

  2PC（两阶段提交）：
    1. Prepare：协调者问所有参与者"能提交吗"
    2. Commit：所有参与者同意 → 协调者发 commit
    ★ 问题：协调者崩溃 → 参与者卡住

  Saga 模式：
    拆成多个本地事务 → 每个有补偿操作
    失败 → 执行已执行的补偿（回滚）
    ★ 最终一致，不阻塞

  ★ 存储系统：避免分布式事务
    单分片事务（数据分片，事务不跨分片）
    → 简单且高效
```

## 7. 时钟与一致性

```
物理时钟不可靠：
  - NTP 同步有误差（毫秒级）
  - 不同机器时钟不同步

逻辑时钟（Lamport Clock）：
  每个事件递增计数器
  → 不依赖物理时钟 → 但只有偏序

混合时钟（HLC）：
  物理时钟 + 逻辑计数
  → 接近物理时间 + 因果顺序

  ★ CockroachDB/YugabyteDB 用 HLC
    → 跨节点事务按 HLC 排序
```

## 8. 一致性级别

```
强一致（Linearizability）：
  读总能看到最新写 → 实现复杂、性能低

顺序一致（Sequential Consistency）：
  所有节点看到的操作顺序一致 → 但不一定实时

因果一致（Causal Consistency）：
  有因果关系的操作顺序一致 → 并发操作无序

最终一致（Eventual Consistency）：
  如果停止写入 → 最终所有副本一致 → AP 系统常用

  ★ 存储系统的选择：
    元数据 → 强一致（Raft）
    数据 → 可选最终一致（性能优先）
    事务 → 强一致（2PC/Quorum）
```
