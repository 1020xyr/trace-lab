# 异常处理与容错

## 1. 失败是常态

```
生产环境会遭遇的失败：
  ├── 网络抖动/超时/分区
  ├── 磁盘坏道/EIO/ENOSPC
  ├── 内存不足/OOM
  ├── 进程崩溃（bug/kill -9）
  ├── 时钟漂移
  ├── 依赖服务不可用
  └── CPU 高负载导致响应慢

  ★ 设计原则：假设所有依赖都会失败
  ★ 优雅降级 > 直接崩溃
```

## 2. 错误分类

```
瞬时错误（transient）→ 可重试
  例：网络抖动、临时过载、连接 reset
  → 重试可能成功

永久错误（permanent）→ 不重试
  例：文件不存在(ENOENT)、权限拒绝(EACCES)、参数错误
  → 重试也不会成功

混合错误 → 需要判断
  例：超时 → 可能是瞬时也可能是永久
  → 看上下文判断
```

## 3. 重试策略

### 指数退避 + 抖动

```c
// 错误的重试（重试风暴）
for (i = 0; i < 5; i++) {
    if (do_request() == 0) break;
    sleep(1);  // 固定间隔
}
// ★ 问题：所有客户端同时重试 → 重试风暴

// 正确的重试（指数退避 + 抖动）
for (i = 0; i < 5; i++) {
    if (do_request() == 0) break;
    int base = 100 * (1 << i);  // 100, 200, 400, 800, 1600ms
    int jitter = rand() % base; // ★ 随机抖动
    usleep((base + jitter) * 1000);
}
// ★ 抖动让重试分散，避免同步重试风暴
```

### 重试上限

```
必须有上限：
  → 网络分区时无限重试 → 资源耗尽
  → 上限 + 最终失败 → 触发降级/告警

重试次数 × 退避时间 = 总等待时间
  例：5 次 × (0.1 + 0.2 + 0.4 + 0.8 + 1.6) = 3.1 秒
  → 3 秒后放弃 → 返回降级结果
```

## 4. 超时设计

```
为什么需要超时：
  → 没有超时 → 一个慢请求占住连接 → 连接池耗尽
  → 雪崩：上游超时 → 上游重试 → 下游更过载

超时层级：
  连接超时（connect timeout）：建立连接
  读超时（read timeout）：读数据
  写超时（write timeout）：写数据
  整体超时（request timeout）：整个请求

  ★ 超时预算（timeout budget）：
    整体超时 = 连接 + 读 + 写 + 处理
    每层超时之和不能超过整体超时
```

## 5. 熔断器（Circuit Breaker）

```
原理：连续失败到阈值 → 短路，不再请求

三态：
  closed（关闭）：正常请求
    ↓ 连续失败 N 次
  open（打开）：★ 不请求，直接返回错误
    ↓ 等待一段时间
  half-open（半开）：★ 放少量请求试探
    ├── 成功 → 回到 closed
    └── 失败 → 回到 open

为什么防雪崩：
  下游故障 → 上游继续请求 → 下游更过载
  → 熔断 → 上游不请求 → 下游有机会恢复
```

## 6. 降级（Degradation）

```
降级策略（按优先级）：
  1. 返回缓存（可能过期但可用）
  2. 返回默认值（空列表/占位符）
  3. 返回简化结果（去掉非核心功能）
  4. 直接报错（最后的手段）

  ★ 降级 vs 直接报错：
    降级 → 用户有结果（可能不完美）
    报错 → 用户无结果
    → 生产优先选择降级
```

## 7. 背压（Backpressure）

```
生产者快于消费者时：

  方式 1：阻塞（blocking）
    生产者阻塞 → 消费者跟上 → 同步
    ★ 适合内存内

  方式 2：丢弃
    丢弃最新/最旧 → 控制队列不无限增长
    ★ 适合可丢弃的数据（监控指标）

  方式 3：缓冲
    增大队列 → 暂时缓冲 → 但不能无限大

  方式 4：反压信号
    生产者收到"慢点"信号 → 降低速率
    ★ TCP 滑动窗口就是这个机制

  ★ 存储系统的反压：
    I/O 队列满了 → 告诉上层"别提交了"
    → 否则内存被 I/O 请求占满
```

## 8. 幂等性

```
幂等：同一个操作执行多次，结果一致

为什么重要：
  重试/重放/至少一次语义 → 都依赖幂等

实现方式：
  唯一 ID + 去重表
    每个请求带唯一 request_id
    服务端记录已处理的 request_id
    重复请求 → 检查去重表 → 返回上次结果

  ★ 存储系统的幂等：
    写请求带版本号
    写入时检查版本 → 旧版本写请求忽略
    → 重试不会导致数据回滚
```

## 9. 信号处理与优雅退出

```c
// SIGTERM 优雅退出
void sigterm_handler(int sig) {
    // 1. 停止 accept（不接新请求）
    close(listen_fd);
    
    // 2. 设置退出标志
    shutdown_flag = 1;
    
    // 3. ★ flush 脏数据到磁盘（存储系统必须！）
    flush_dirty_data();
    fsync(data_fd);
    
    // 4. 等待 worker 处理完已有请求
    while (active_requests > 0 && !timeout)
        usleep(10000);
    
    // 5. 退出
    exit(0);
}

// ★ SIGKILL 无法捕获
//   → 必须设计成"随时被 kill -9 后能恢复"
//   → 依赖 WAL/journal 恢复
```

## 10. FUSE 特有的异常处理

```
FUSE daemon 崩溃：
  内核 FUSE 模块 → 挂起请求返回错误
  
  客户端视角：
    open/read/write → ENOTCONN 或 EIO
    
  处理策略：
    1. 客户端检测到 ENOTCONN → 等待 + 重试
    2. 重试期间 daemon 可能已重启
    3. ★ 需要幂等性（重试不导致重复写入）

  挂载点 stale：
    访问 → Transport endpoint is not connected
    → 监控脚本检测 → umount + remount
    → 或自动重启 daemon
```
